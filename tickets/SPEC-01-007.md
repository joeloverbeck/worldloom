# SPEC-01-007: Index CRUD (nodes, edges, FTS5 rebuild) + file-enumeration inventory

**Status**: PENDING
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — adds `tools/world-index/src/index/nodes.ts`, `tools/world-index/src/index/edges.ts`, `tools/world-index/src/index/fts.ts`, and `tools/world-index/src/enumerate.ts`. These are the SQLite-write layer the CLI commands will compose in SPEC-01-008.
**Deps**: SPEC-01-002 (schema + `openIndex`), SPEC-01-003 (row-shape types)

## Problem

All four parser passes produce row candidates (nodes, edges, entity mentions, anchor checksums, validation results). This ticket wires those rows into the SQLite DB via three concerns:
- **File enumeration** — which files under `worlds/<slug>/` count as indexable (the explicit include/exclude list added to SPEC-01 during reassessment).
- **Node/edge CRUD** — `insertNodes`, `deleteNodesByFile`, `insertEdges`, `resolveUnresolvedEdges`, `insertEntityMentions`, `insertAnchorChecksums`, `insertValidationResults`.
- **FTS5 maintenance** — the triggers from SPEC-01-002 do the per-row sync automatically; this module adds the >10% churn rebuild optimization (`INSERT INTO fts_nodes(fts_nodes) VALUES('rebuild')`) as an end-of-sync hook.

The file-enumeration inventory is critical: silently skipping `character-proposals/` (as the pre-reassessment spec did) would leave 7 NCP cards out of the index. This ticket implements the enumeration as a first-class module.

## Assumption Reassessment (2026-04-22)

1. SPEC-01 §File-enumeration inventory (post-reassessment) enumerates the exact paths to include: world-level `.md`, `adjudications/*.md`, `characters/*.md`, `diegetic-artifacts/*.md`, `proposals/*.md`, `proposals/batches/*.md`, `character-proposals/*.md`, `character-proposals/batches/*.md`, `audits/*.md`, `audits/*/retcon-proposals/*.md`. Excluded: `_index/`, `_source/`, dotfiles, all `INDEX.md` files. Anything else yields a `validation_results` warn row.
2. SPEC-01 §Public contract (verified in session): sibling packages read `world.db` directly; this CRUD module is INTERNAL to `tools/world-index`. It is not exported from the package and not consumed by `world-mcp` / `patch-engine` / `validators`. Those packages run their own SQL.
3. Cross-artifact boundary: this module is the sole writer to `nodes`, `edges`, `entity_mentions`, `file_versions`, `anchor_checksums`, `validation_results`. It is NOT the sole writer to `fts_nodes` — the triggers from SPEC-01-002 are. The module exposes a `rebuildFtsIndex(db)` helper for the >10% churn path. Grep-proof in Acceptance invariants.
4. FOUNDATIONS principle under audit: **Rule 6 (No Silent Retcons)** — deleting a node cascades to its edges and its entity_mentions; the pattern MUST NOT silently delete a `change_log_entry` node while leaving orphaned `affected_fact` edges pointing to deleted facts. Every cascade is explicit SQL. FKs with `ON DELETE` are not used because SQLite FK cascades would dilute the explicit-deletion audit trail; deletions are explicit `DELETE FROM edges WHERE source_node_id IN (...)`.
5. Schema extension posture: this ticket ships purely additive code; no migration changes. The `validation_results` warn-row recording path is the structural hook SPEC-04 will consume.

## Architecture Check

1. Three small focused modules (`nodes.ts`, `edges.ts`, `fts.ts`) plus a fourth `enumerate.ts` for file enumeration. Alternative — single `crud.ts` — was rejected because file enumeration is a pure function over the filesystem (no DB handle) and deserves its own module for testability.
2. `enumerate.ts` returns a `FileEnumeration` object with two arrays: `indexable: string[]` (relative paths) + `unexpected: string[]` (paths that matched no rule — emitted as warn rows). Explicit `unexpected` list means "silent skipping" is impossible; every file is accounted for.
3. Batch inserts use prepared statements + `db.transaction(...)` to keep large-ledger ingest fast (~500× faster than per-row inserts for animalia's ~5000-node index). No backwards-compatibility shims. Transactions are synchronous because `better-sqlite3` is synchronous.
4. Explicit cascade-delete in `deleteNodesByFile` (delete entity_mentions → edges → nodes) keeps the order reviewable and auditable. Rule-6 preservation over ergonomic FK cascades.

## Verification Layers

1. `enumerate('worlds/animalia')` returns an `indexable` list containing the 13 mandatory files + every file under the seven expected subdirectories, and an empty `unexpected` list → structural enumeration test.
2. Adding an unexpected file (e.g., `worlds/animalia/notes.md`) under the world root puts it in `unexpected` AND emits one warn `validation_results` row at sync time → unexpected-path discipline.
3. `insertNodes` writing 100 rows inside `db.transaction(...)` completes in <50ms on commodity hardware → batch-insert perf.
4. `deleteNodesByFile('worlds/fixture/FOO.md')` removes associated nodes + edges + entity_mentions; an independent `SELECT` on `edges` returns zero rows whose source or target was removed → explicit cascade.
5. `resolveUnresolvedEdges(db)` after inserting a previously-missing target node populates `target_node_id` and clears `target_unresolved_ref` on every matching edge → re-resolution pass.
6. `rebuildFtsIndex(db)` runs `INSERT INTO fts_nodes(fts_nodes) VALUES('rebuild')` and does not throw on a fresh DB → FTS5-maintenance optimization.
7. FOUNDATIONS alignment: no module under `src/index/` writes to `fts_nodes` directly (trigger-only sync invariant from SPEC-01-002) → grep-proof.

## What to Change

### 1. `tools/world-index/src/enumerate.ts`

Exports:
- `interface FileEnumeration { indexable: string[]; unexpected: string[]; }`
- `enumerate(worldRoot: string): FileEnumeration` — uses `fs.readdirSync({ recursive: true })` (Node ≥20) or a manual recursive walk; applies the include-list + exclude-list from SPEC-01 §File-enumeration inventory.

Exclusions take precedence over inclusions. An `INDEX.md` anywhere is excluded. A file matching neither include nor exclude is `unexpected`.

### 2. `tools/world-index/src/index/nodes.ts`

Exports:
- `insertNodes(db: Database, rows: NodeRow[]): void` — prepared-statement batch insert inside `db.transaction`.
- `deleteNodesByFile(db: Database, worldSlug: string, filePath: string): void` — cascades explicitly: delete entity_mentions of affected nodes → delete edges where source or target is affected → delete nodes.
- `insertAnchorChecksums(db: Database, rows: AnchorChecksumRow[]): void`.
- `insertEntityMentions(db: Database, rows: EntityMentionRow[]): void`.
- `insertValidationResults(db: Database, rows: ValidationResultRow[]): void`.

### 3. `tools/world-index/src/index/edges.ts`

Exports:
- `insertEdges(db: Database, rows: EdgeRow[]): void`.
- `resolveUnresolvedEdges(db: Database): number` — returns number of edges resolved; implements the spec's "for each `edges.target_unresolved_ref`, look up if a node now exists" pass.
- `dedupeEdges(rows: EdgeRow[]): EdgeRow[]` — in-memory dedupe of `(source, target, edge_type)` triples before insert; typed edges take precedence over `mentions_id` candidates.

### 4. `tools/world-index/src/index/fts.ts`

Exports:
- `rebuildFtsIndex(db: Database): void` — `db.exec("INSERT INTO fts_nodes(fts_nodes) VALUES('rebuild')")`.
- `shouldRebuildFts(changedNodeCount: number, totalNodeCount: number): boolean` — returns `totalNodeCount > 0 && changedNodeCount / totalNodeCount > 0.1`.

No direct writes to `fts_nodes` — all sync happens via the triggers.

### 5. `tools/world-index/src/index/file-versions.ts`

Exports:
- `getFileVersion(db, worldSlug, filePath): string | null`.
- `upsertFileVersion(db, worldSlug, filePath, contentHash): void`.

## Acceptance Criteria

### Tests That Must Pass

1. `enumerate('<fixture-world>')` returns every expected path in `indexable` (13 world files + subdirectory files) and empty `unexpected`.
2. Adding `<fixture-world>/scratch.md` causes `scratch.md` to appear in `unexpected`.
3. Adding `<fixture-world>/.hidden.md` is excluded (dotfile) — not in `indexable` nor `unexpected`.
4. `insertNodes(db, 100 rows)` + `SELECT COUNT(*) FROM nodes` → 100.
5. Following (4) with `deleteNodesByFile(db, slug, filePath)` removes every node + their edges + their entity_mentions — a `SELECT COUNT(*)` on each of the three tables returns 0 rows sourced from that file.
6. Inserting an edge with `target_unresolved_ref='CF-0999'`, then inserting a node with `node_id='CF-0999'`, then calling `resolveUnresolvedEdges` — the edge's `target_node_id='CF-0999'` and `target_unresolved_ref` is NULL.
7. `shouldRebuildFts(150, 1000)` returns `true`; `shouldRebuildFts(50, 1000)` returns `false`.

### Invariants

1. No code under `src/index/` (except `nodes.ts`'s cascade-delete path) issues `INSERT`/`UPDATE`/`DELETE` against `fts_nodes` — grep-proof against `fts_nodes` SQL outside `src/schema/migrations/` and `src/index/fts.ts`'s `rebuild` branch.
2. Every batch insert runs inside `db.transaction(...)` — grep-proof that every `db.prepare(...).run(...)` call in `src/index/` happens inside a `db.transaction(() => {...})` closure.
3. `enumerate` never returns a path matching both `indexable` and `unexpected` — idempotence check in tests.
