# SPEC-01-010: Patch vulnerable direct dependency `yaml` in `tools/world-index/`

**Status**: PENDING
**Priority**: HIGH
**Effort**: Small
**Engine Changes**: Yes — updates `tools/world-index/package.json` and `tools/world-index/package-lock.json` to a patched `yaml` release; no behavior change intended.
**Deps**: SPEC-01-001 (package scaffold)

## Problem

`SPEC-01-001` pinned `yaml@2.8.1` in the new `tools/world-index/` scaffold. Post-ticket review ran `npm audit --json` and found a direct moderate-severity advisory on `yaml` affecting `>=2.0.0 <2.8.3` (`GHSA-48c2-rrv3-qjmp`, stack overflow via deeply nested YAML collections). Leaving the scaffold on a known vulnerable direct dependency weakens the package baseline before parser work lands.

## Assumption Reassessment (2026-04-22)

1. Live `tools/world-index/package.json` pins `yaml` to `2.8.1`, and live `tools/world-index/package-lock.json` resolves the same version as a direct dependency.
2. `npm audit --json` run in `tools/world-index/` during post-ticket review reported advisory `GHSA-48c2-rrv3-qjmp` on direct dependency `yaml`, with `fixAvailable.version = 2.8.3` and `isSemVerMajor = false`.
3. Owned boundary: this ticket is limited to the package dependency contract for `tools/world-index/`; it does not own parser behavior, schema work, or command implementation.
4. Mismatch corrected by follow-up rather than hot-editing the archived ticket: SPEC-01-001 truthfully captured the scaffold that landed, but review established that the pinned `yaml` version is no longer an acceptable steady-state baseline.

## Architecture Check

1. Bumping to the patched non-breaking release keeps the deterministic pinned-dependency contract from SPEC-01-001 while removing a known vulnerability. This is cleaner than loosening the range or deferring package hygiene to a later parser ticket.
2. No backwards-compatibility aliasing or shims are introduced; this is a direct dependency/version correction only.

## Verification Layers

1. Direct dependency contract -> codebase grep-proof on `tools/world-index/package.json` and `tools/world-index/package-lock.json`.
2. Security advisory closure -> targeted tool command `npm audit --json` reports zero vulnerabilities after the bump.
3. Scaffold integrity -> targeted tool commands `npm run build` and `npm test` still pass after the lockfile refresh.
4. Single-package boundary -> manual review of touched files confirms no production source changes outside dependency metadata.

## What to Change

### 1. Update the pinned `yaml` version

Change `tools/world-index/package.json` from `yaml: "2.8.1"` to `yaml: "2.8.3"`.

### 2. Refresh the lockfile

Regenerate `tools/world-index/package-lock.json` so the resolved direct dependency and integrity hashes match `yaml@2.8.3`.
