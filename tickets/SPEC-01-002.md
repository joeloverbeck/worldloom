# SPEC-01-002: SQLite migration 001 + connection opener + index_version sidecar

**Status**: PENDING
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — adds `tools/world-index/src/schema/migrations/001_initial.sql` + `tools/world-index/src/index/open.ts`; produces `worlds/<slug>/_index/world.db` + `worlds/<slug>/_index/index_version.txt` at runtime.
**Deps**: SPEC-01-001 (package scaffold)

## Problem

SPEC-01 specifies an 8-table SQLite schema (`nodes`, `edges`, `entity_mentions`, `file_versions`, `anchor_checksums`, `fts_nodes`, `summaries`, `validation_results`) with external-content FTS5 kept coherent via three triggers (`nodes_ai`, `nodes_ad`, `nodes_au`). Migrations are forward-only; schema-version mismatch exits `1` and instructs the user to rebuild from scratch. Every downstream ticket (nodes CRUD, parser-pass writes, CLI commands) depends on a working `openIndex(worldSlug)` call that creates the `_index/` directory, runs the migration on a fresh DB, and writes a sidecar `index_version.txt` recording the applied version.

## Assumption Reassessment (2026-04-22)

1. SPEC-01 §SQLite schema enumerates all 8 tables with exact column names, PK/FK constraints, and the two `idx_nodes_*` indexes + two `idx_edges_*` indexes. The FTS5 virtual table uses `content='nodes'` + `content_rowid='rowid'` (external-content). The three triggers are defined verbatim in SPEC-01 and copy directly into `001_initial.sql`.
2. SPEC-01 §Index location: DB lives at `worlds/<world-slug>/_index/world.db`; companion `worlds/<world-slug>/_index/index_version.txt` records the applied migration number. `.gitignore` already excludes `worlds/*/_index/` (verified in root `.gitignore`).
3. Cross-artifact boundary: the migration SQL is the **public schema contract** consumed by SPEC-01-007's CRUD layer AND (via `world.db` black-box) by `tools/world-mcp`, `tools/patch-engine`, `tools/validators`. Per SPEC-01 §Public contract, those packages open the DB directly via their own `better-sqlite3` connections. Schema drift without a migration bump is a Rule-6 silent-retcon risk; migration ordering is forward-only.
4. `better-sqlite3` executes multi-statement SQL via `db.exec(sql)` (synchronous); FTS5 support is compiled into the bundled libsqlite, no runtime compile flags needed.

## Architecture Check

1. Co-locating the schema in a single `001_initial.sql` file (rather than splitting per-table into multiple migrations) is cleaner for the Phase 1 baseline: the schema is a frozen contract the spec froze, and a single file reads as the canonical "what the DB looks like" reference. Later schema additions ship as `002_<name>.sql`, keeping the forward-only discipline explicit.
2. Trigger-based FTS5 sync (per SPEC-01 Q3 resolution) avoids scattering `INSERT INTO fts_nodes` calls across every CRUD call site. Triggers are declarative, run inside every transaction, and are cross-connection-safe — any future reader that writes to `nodes` gets FTS5 sync for free. No backwards-compatibility shims.
3. `openIndex(worldSlug)` is module-internal to `tools/world-index`; it is NOT exported from the package. Sibling packages open `world.db` themselves per SPEC-01 §Public contract. This preserves the pure-producer posture.

## Verification Layers

1. Fresh `openIndex('fixture-world')` creates all 8 tables + 4 explicit indexes + 3 triggers → `sqlite_master` SELECT proves object count + names (SQL grep-proof).
2. Inserting a nodes row with `body='foo bar'` and querying `fts_nodes MATCH 'foo'` returns the row → FTS5 AFTER-INSERT trigger wiring is correct (schema validation).
3. `DELETE FROM nodes WHERE node_id = 'x'` followed by `fts_nodes MATCH` returns no rows → AFTER-DELETE trigger wiring is correct (schema validation).
4. `UPDATE nodes SET body='new' WHERE node_id='x'` removes old FTS terms and inserts new ones → AFTER-UPDATE trigger is correct (schema validation).
5. Opening an existing `world.db` whose `index_version.txt` doesn't match `CURRENT_INDEX_VERSION` exits 1 with the SPEC-01-mandated message → forward-only migration discipline (CLI exit-code check).
6. FOUNDATIONS alignment: the schema covers all 19 node types enumerated in SPEC-01 via the single string `nodes.node_type` column (no per-type tables). Rule 6 audit trail is preserved via `modification_history` + `change_log_entry` nodes + `edges.edge_type` values `modified_by` / `originates_in` / `patched_by` / `clarified_by` — verified at schema-design time.

## What to Change

### 1. `tools/world-index/src/schema/migrations/001_initial.sql`

Exact DDL from SPEC-01 §SQLite schema — all 8 tables, 4 indexes, FTS5 virtual table, and 3 triggers (`nodes_ai`, `nodes_ad`, `nodes_au`). No field renames, no column additions, no ordering changes relative to the spec. Wrap in a `BEGIN TRANSACTION; ... COMMIT;` block so partial migration failure leaves no half-applied schema.

### 2. `tools/world-index/src/index/open.ts`

Exports a module-internal `openIndex(worldRoot: string, worldSlug: string): Database` function:

1. Resolves `<worldRoot>/worlds/<worldSlug>/_index/` and creates the directory via `fs.mkdirSync({ recursive: true })` if absent.
2. Opens `better-sqlite3` connection against `_index/world.db`.
3. Reads `_index/index_version.txt`. If absent (fresh DB), runs `001_initial.sql` via `db.exec(readFileSync(...))`, then writes `'1\n'` to `index_version.txt`.
4. If present and the recorded version differs from `CURRENT_INDEX_VERSION = 1`, throws a `SchemaVersionMismatchError` (caller translates to exit code `1`).
5. Enables `PRAGMA journal_mode=WAL`, `PRAGMA synchronous=NORMAL`, `PRAGMA foreign_keys=ON` for write-path robustness + referential-integrity enforcement (spec uses `FOREIGN KEY` clauses in the schema).

Export `openIndex` from `src/index/open.ts` but NOT from `src/cli.ts` or any re-export barrel. No `index.ts` barrel file exists in the package (per SPEC-01-001's no-library-surface invariant).

### 3. Constant for migration version

Add `export const CURRENT_INDEX_VERSION = 1` in `src/schema/version.ts`. Future migration tickets bump this constant in lockstep with a new migration file.

### 4. Fixture SQL for tests

Create `tools/world-index/tests/fixtures/empty-db-init.sql` — a short SQL script used by the schema test to seed minimal `nodes` / `fts_nodes` rows for trigger verification.

## Acceptance Criteria

### Tests That Must Pass

1. `openIndex('/tmp/test-root', 'fixture-world')` on a non-existent path creates `/tmp/test-root/worlds/fixture-world/_index/world.db` + `index_version.txt` containing `1\n`.
2. Querying `SELECT name FROM sqlite_master WHERE type='table'` returns exactly `{nodes, edges, entity_mentions, file_versions, anchor_checksums, fts_nodes, fts_nodes_data, fts_nodes_idx, fts_nodes_docsize, fts_nodes_config, summaries, validation_results}` (FTS5 shadow tables are automatic).
3. Querying `SELECT name FROM sqlite_master WHERE type='trigger'` returns `{nodes_ai, nodes_ad, nodes_au}`.
4. Insert → `fts_nodes MATCH` → row found; delete → `fts_nodes MATCH` → no row; update → old-term gone, new-term findable (three trigger tests).
5. Opening an existing DB whose `index_version.txt` contains `2\n` raises `SchemaVersionMismatchError`.
6. `PRAGMA foreign_keys` returns `1` after `openIndex`.

### Invariants

1. Migration 001 is append-only relative to the spec's SQL — any future schema change ships as `002_*.sql` + `CURRENT_INDEX_VERSION` bump (SPEC-01 §Error model: migrations are forward-only in Phase 1).
2. `openIndex` is never re-exported from the package root — sibling packages open `world.db` directly via their own `better-sqlite3` (SPEC-01 §Public contract; SPEC-01-001 acceptance invariant 1).
3. FTS5 coherence is maintained only via the three triggers; no CRUD path writes to `fts_nodes` directly (enforced by grep-proof in later tickets).
