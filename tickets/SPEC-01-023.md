## Problem

Fresh rebuild audit on `2026-04-22` shows `worlds/animalia/_index/world.db` is structurally clean but still semantically noisy on a narrow `named_entity` seam. After `npm run build`, `node tools/world-index/dist/src/cli.js build animalia`, and `node tools/world-index/dist/src/cli.js verify animalia`, the rebuilt DB still persists inline section/open-question reference labels as first-class entities:

- `Trade Flows` (`17` mentions)
- `Inequality Patterns` (`20` mentions)
- `Ruin Ownership` (`11` mentions)
- `Breakage Point` (`8` mentions)
- `Tier Finder` (`7` mentions)
- `Fee Wage Schedule` (`7` mentions)
- `Count Specifics` (`4` mentions)

These are not stale checked-in `_index` residue. They reproduce after fresh rebuild, and their source rows are inline references to document headings or deferred-question labels rather than world entities. That weakens the machine-facing semantic surface that downstream retrieval or audit consumers would query directly from SQLite.

2. No backwards-compatibility aliasing or parallel "filtered entity" layer should be introduced. The canonical `named_entity` surface itself should become trustworthy.

## What to Change

### 1. Retune the heuristic boundary for inline document-reference labels

Teach `tools/world-index/src/parse/entities.ts` to reject section/open-question cross-reference labels and truncated heading fragments that arise from prose like `ECONOMY §Trade Flows`, `OPEN_QUESTIONS §Ruin Ownership and Jurisdiction`, and similar inline document citations.

### 2. Extend stoplist or fragment rules only for this bounded seam

Add the minimum extractor-side filtering needed for the reproduced label class (`Trade Flows`, `Inequality Patterns`, `Ruin Ownership`, `Breakage Point`, `Tier Finder`, `Fee Wage Schedule`, `Count Specifics`) without widening into unrelated phrase-fragment cleanup.

### 3. Add rebuilt-`animalia` regression coverage

Encode the audited false positives directly in package tests so future rebuilds cannot silently repollute the semantic surface with document-reference labels.
