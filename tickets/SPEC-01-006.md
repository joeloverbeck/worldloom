# SPEC-01-006: Pass 3 world-semantic extraction + entity scanner

**Status**: PENDING
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — adds `tools/world-index/src/parse/semantic.ts` and `tools/world-index/src/parse/entities.ts`. Emits the 13 typed edge categories (post-reassessment) and the `named_entity` virtual nodes that the retrieval surface depends on.
**Deps**: `tools/world-index/src/schema/types.ts` (live types including `EdgeType` union), SPEC-01-005 (Pass 2 YAML-backed node rows)

## Problem

Pass 3 is where "12,111 lines of markdown" becomes "structured graph the retrieval layer queries." Three tasks fold together: (a) scan every node body with the structured-ID regex to produce candidate edges; (b) walk typed YAML objects from Pass 2 to emit edges from specific fields (`source_basis.derived_from`, `modification_history[].change_id`, `affected_fact_ids`, `originating_cf` on CH nodes, `pre_figured_by`); (c) extract attribution-comment edges (`<!-- added by CF-NNNN -->`) from prose nodes under the deterministic "nearest-enclosing structural node" source-node rule; (d) scan for named entities via ONTOLOGY.md registry + capitalized-multiword heuristic. Getting (c) wrong — conflating YAML-edge and attribution-edge semantics — was a HIGH finding in the reassessment; this ticket encodes the corrected post-reassessment separation.

## Assumption Reassessment (2026-04-22)

1. SPEC-01 §Parser pipeline Pass 3 (post-reassessment) now specifies:
   - ID reference regex: `/\b(CF|CH|PA|M|DA|CHAR|PR|BATCH|NCP|NCB|AU|RP)-\d+\b/g`.
   - YAML-field edges keyed from typed CF/CH objects; `originating_cf` is read **only** from the `change_log_entry` node, never from the embedded `modification_history` copies, to avoid duplicate edges.
   - Attribution-comment edges: `patched_by` (prose→CF) and `clarified_by` (prose→CH) — distinct from `modified_by`/`originates_in` (the YAML-derived edges). The attribution comment's verb is informational; the target-id prefix selects the edge type.
   - Source-node rule for attribution comments: "nearest enclosing structural node — `subsection` if one exists between the comment and the nearest `section`, else `section`; for comments inside GFM table cells, the source is the containing `subsection` (or `section` if no subsection). Comments inside fenced code blocks are ignored."
2. `entities.ts` uses `worlds/<slug>/ONTOLOGY.md` as the entity registry (animalia ONTOLOGY.md is 81 lines — verified) plus a capitalized-multiword heuristic. Pass 3 is tunable; false positives are acceptable at Phase 1 since SPEC-04's `world-validate` will report drift.
3. Cross-artifact boundary: the `EdgeRow` emissions flow into `SPEC-01-007` edges-CRUD with `target_unresolved_ref` populated when the target node isn't indexed yet; the sync algorithm (`SPEC-01-008`) re-resolves after full-pass. `named_entity` virtual nodes are emitted as `NodeRow` with `node_type='named_entity'` + `node_id='entity:<canonical-slug>'`; their `entity_mentions` backing rows are written by the same pass. Three consumer surfaces (retrieval backlink expansion, validator-rule-12 trace-register coverage, patch engine's anchor localization) depend on this pass's emissions.
4. FOUNDATIONS principle under audit: **Rule 6 (No Silent Retcons)** — the `patched_by` / `clarified_by` disambiguation (enforced at this layer) preserves the Rule-6 audit trail by separating "who modified this fact" (`modified_by`) from "who patched this prose" (`patched_by`). **Rule 7 (Preserve Mystery Deliberately)** — the `firewall_for` edge (MR→CF) is the structural enforcement surface retrieval uses to block MR-resolving queries; this pass emits those edges but does not evaluate them.
5. `pre_figured_by` edge: CF YAML field `pre_figured_by:` — array of DA-NNNN / CHAR-NNNN ids. Currently zero occurrences in animalia (forward-looking per SPEC-01).

## Architecture Check

1. Splitting Pass 3 into `semantic.ts` (ID scan + edge extraction) and `entities.ts` (entity scanner) — rather than one mega-module — isolates the tunable-heuristic surface from the deterministic-regex surface. Entity extraction is expected to evolve post-launch; isolating it makes the tuning surface obvious.
2. The source-node rule for attribution comments is a **deterministic** function of the mdast position + the heading-path stack; it never depends on node-body content. That keeps the parser idempotent and makes debugging "why is this edge sourced here" a matter of reading the rule, not tracing author intent. No backwards-compatibility shims.
3. The canonical-slug generator for `named_entity` ids (e.g., `entity:brinewick`) uses NFC-normalized + lowercased + non-alphanumeric→hyphen transformation, with collision fallback to `entity:<slug>-<short-hash>`. Alternative — using the registry's literal name as the id — was rejected because author-register variation ("the Brinewick", "brinewick") should collapse into one entity.

## Verification Layers

1. A fixture containing the literal text `See also CF-0021 and M-2.` produces exactly two unresolved candidate edges from the ID scan (one per match) → regex extraction.
2. A CF YAML block with `modification_history: [{ change_id: CH-0014, originating_cf: CF-0039 }]` produces one `modified_by` edge (source = this CF, target = CH-0014). Reading the `originating_cf` of the embedded modification_history entry does NOT produce an `originates_in` edge here — only the CH node's own `originating_cf` field does (deduplication rule).

4. A GFM table cell under section `## Magical Effects` (no subsection) containing `<!-- added by CF-0039 -->` produces a `patched_by` edge whose source is the section-level node → fallback to `section` when no `subsection` exists.
5. An ONTOLOGY.md entry `- Brinewick (polity, coastal)` plus a paragraph containing `The watchmen of Brinewick reported...` produces one `mentions_entity` edge to `entity:brinewick` → registry + case-insensitive matching.
6. FOUNDATIONS alignment: the edge-type set produced by this pass is exactly the 13 in SPEC-01 §Edge types (post-reassessment) → grep cross-check `EDGE_TYPES` constant in `tools/world-index/src/schema/types.ts`.

## What to Change

### 1. `tools/world-index/src/parse/semantic.ts`

Exports:
- `extractSemanticEdges(tree: Root, lines: string[], filePath: string, yamlNodes: NodeRow[], proseNodes: NodeRow[]): { edges: EdgeRow[]; parseIssues: ValidationResultRow[] }`.

Algorithm:
1. **ID scan** — iterate every prose node body and every YAML node body; match the regex `/\b(CF|CH|PA|M|DA|CHAR|PR|BATCH|NCP|NCB|AU|RP)-\d+\b/g`; each match becomes a generic candidate edge `(source_node_id=<this node>, target_unresolved_ref=<match>, edge_type='mentions_id')` — **except** when the same `(source, target)` pair will already be emitted by a typed YAML-field rule below; typed rules take precedence over the generic scan. NOTE: `mentions_id` is a candidate edge only; SPEC-01-007's CRUD layer deduplicates against typed edges before insert.
2. **YAML-field edge extraction** — for each CF `NodeRow` whose body parsed successfully in Pass 2:
   - `source_basis.derived_from` → `derived_from` edges.
   - `required_world_updates` → `required_world_update` edges (target = synthetic file-path node id; file-path nodes are section nodes in the named domain file, resolved by matching the file path).
   - `modification_history[].change_id` → `modified_by` edges.
   - `pre_figured_by` → `pre_figured_by` edges.
   - `applies_to` → `applies_to` edges (when the CF YAML carries an `applies_to:` MR-id list; open-ended per reassessment).
   - CH `NodeRow` → `affected_fact_ids[]` → `affected_fact` edges; `originating_cf` → `originates_in` edge (source = this CH node).
   - MR `mystery_reserve_entry` node → `firewall_for` edges (parse the MR body for `firewall:` or similar structured markers; if the MR is prose-only, skip — document this limitation in code comment).
3. **Attribution-comment extraction** — for every `html` node in the mdast with position + matching `<!--\s*(added|clarified|modified)\s+by\s+(CF|CH)-(\d+)\s*-->`:
   - Walk the heading-path ancestor stack (built during Pass 1 / shared via the caller) to the nearest `subsection` (depth ≥ 3) or `section` (depth = 2); select the deepest. This is the **source node id**.
   - Target is `<prefix>-<number>` from the regex; target type selects edge type: CF → `patched_by`, CH → `clarified_by`.
   - Skip attribution comments whose containing mdast ancestor is a `code` node (fenced code block).
4. Emit `ValidationResultRow` warnings for: attribution comment with a prefix that isn't `CF` or `CH` (malformed attribution); attribution comment whose target ID doesn't resolve at sync-end-unresolved-pass time (flag as `unresolved_attribution_target`).

### 2. `tools/world-index/src/parse/entities.ts`

Exports:
- `extractEntities(tree: Root, proseNodes: NodeRow[], ontologyRegistry: EntityRegistry): { entityNodes: NodeRow[]; mentions: EntityMentionRow[] }`.
- `loadOntologyRegistry(ontologyPath: string): EntityRegistry` — parses `worlds/<slug>/ONTOLOGY.md` into a case-folded lookup of `{ canonicalName, kind }`.

Algorithm:
1. For each registered entity, case-insensitive substring scan of every prose node body; record matches as `EntityMentionRow` with `entity_name` = canonical form.
2. For each prose node body, apply the capitalized-multiword heuristic: regex `/\b([A-Z][a-z]+(?:\s+[A-Z][a-z]+){1,3})\b/g` — candidate names. Filter against a stop list (common patterns like "The Watchman", "A Certain", proper-noun-styled sentence starts). Residual candidates become `entity:<canonical-slug>` virtual nodes + corresponding mentions.
3. Virtual `named_entity` NodeRows: one per unique canonical entity, `body` = "Canonical name: ... | Kind: ... | Mentions: N". `content_hash` is computed over the body via `contentHashForProse` (deterministic given the same mentions set).

### 3. Stop list asset

Add `tools/world-index/src/parse/stoplist.ts` containing the initial capitalized-multiword stop list: {"The Maker", "The Makers", "A Certain", "Once Upon", ...}. Start conservative; tune via `world-validate` feedback in SPEC-04.

3. `modified_by` and `originates_in` are emitted only from typed YAML parses — never from attribution comments (grep-proof: attribution-comment branch of `semantic.ts` does not write those two edge types).
