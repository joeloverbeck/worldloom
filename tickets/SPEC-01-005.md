# SPEC-01-005: Pass 2 YAML extraction + yaml_parse_integrity recording

**Status**: PENDING
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — adds `tools/world-index/src/parse/yaml.ts`. Introduces the permissive-unknown-field contract the rest of the pipeline relies on and produces the YAML-backed node rows the semantic pass will cross-reference.
**Deps**: `tools/world-index/src/schema/types.ts` (live row-shape/type authority), SPEC-01-004 (Pass 1 mdast + `contentHashForYaml`)

## Problem

`CANON_LEDGER.md` stores every Canon Fact Record and Change Log Entry as fenced `yaml` blocks inside one of two sections (`## Canon Fact Records`, `## Change Log`). Pass 2 walks the Pass 1 mdast, locates each fenced YAML block, parses it, validates the parsed shape against `CanonFactRecord` / `ChangeLogEntry` / `ModificationHistoryEntry` interfaces from `tools/world-index/src/schema/types.ts`, and emits a typed node record plus a canonical content hash. Parse failures must never kill the build — they record a `yaml_parse_integrity` row in `validation_results` (SPEC-04's consumer) and emit a partial node so retrieval still surfaces the broken block for debugging. Unknown top-level fields must be preserved to absorb SPEC-09's future schema additions.

## Assumption Reassessment (2026-04-22)

1. SPEC-01 §Parser pipeline Pass 2 now reads: *"walks `code` nodes with `lang === 'yaml'`; parses each with the `yaml` package; validates against `CanonFactRecord` / `ChangeLogEntry` / `ModificationHistoryEntry` TypeScript interfaces... The parser is **permissive about unknown top-level fields** — unknown fields are preserved inside the node's `body` text for downstream validators."* This ticket implements that posture.
2. SPEC-01 §Risks (verified in session): CF-vs-CH discrimination relies on the parent section heading (`## Canon Fact Records` vs `## Change Log`). Actual animalia ledger has both headings at the expected location (grepped in reassessment: `^## Canon Fact Records` line 7, `^## Change Log` line 6365). `create-base-world` mandates these exact headings; drift is flagged as a `yaml_parse_integrity` warn-severity entry.
3. Cross-artifact boundary: this pass consumes Pass 1 mdast + `contentHashForYaml` from SPEC-01-004, and produces `NodeRow` candidates typed per `tools/world-index/src/schema/types.ts` that SPEC-01-007's CRUD layer writes. It also emits `ValidationResultRow` candidates for the `validation_results` table. SPEC-01-006 (Pass 3) consumes the typed CF/CH objects to extract YAML-field edges.
4. FOUNDATIONS principle under audit: **Rule 1 (No Floating Facts)** — every CF YAML record must carry `statement`, `scope`, `prerequisites`, `domains_affected`, etc. Pass 2 does NOT enforce Rule 1 at parse time (that's SPEC-04's validator-rule-1 job); it just records what the YAML says. But Rule 1 shapes the `CanonFactRecord` interface: required fields are required, unknown fields are allowed via index signature. **Rule 5 (No Consequence Evasion)**: the permissive-unknown posture is the forward-compat consequence handling for SPEC-09's CF schema extension.
5. Schema extension posture: the `yaml` package's `parse` function returns `unknown`; Pass 2 casts via a `narrowToCanonFactRecord(candidate): CanonFactRecord | null` runtime-check that validates only the **required** fields — `id`, `title`, `status`, `type`, `statement` — and trusts the index signature for everything else. If a required field is absent, emit a `yaml_parse_integrity` warn row and still produce a partial node (the raw YAML text stays in `body`).

## Architecture Check

1. Separating "parse YAML" from "validate against interface" from "emit node row" (three small functions in one module) keeps each concern testable. Fusing them into one function obscures the parse-failure path.
2. Soft validation — required-fields-only, unknown-fields-preserved — is cleaner than the closed-shape alternative because a closed shape would either reject SPEC-09 CFs or force a schema-version branch inside every downstream consumer. The permissive posture pushes version awareness to SPEC-04 validators where it belongs.
3. The CF/CH discriminator uses the ancestor-section heading (walked from the `code` node's parent chain to the nearest `^## ` header). Alternative — prefix-on-the-id discrimination — was considered and rejected: `CF-0001` and `CH-0001` both start with uppercase letters and identically-shaped numerals, which makes prefix detection identical in cost but less robust to future prefix additions. Section-ancestor is also what `create-base-world` commits to via mandatory section headings.

1. A well-formed CF YAML block in `fixture-ledger.md` produces a `NodeRow` with `node_type='canon_fact_record'` + `node_id='CF-NNNN'` + `content_hash = contentHashForYaml(parsed)` → unit schema check.

5. A fenced YAML block inside the `## Canon Fact Records` section is typed as `canon_fact_record`; the identical-content block under `## Change Log` is typed as `change_log_entry` → section-ancestor discriminator.
6. FOUNDATIONS alignment: field names in the validation code match `docs/FOUNDATIONS.md` §Canon Fact Record Schema exactly → grep-proof against the FOUNDATIONS field list.

## What to Change

### 1. `tools/world-index/src/parse/yaml.ts`

Exports:
- `extractYamlNodes(tree: Root, lines: string[], filePath: string): { nodes: NodeRow[]; parseIssues: ValidationResultRow[] }`.
- Internal helpers: `findContainingSection(codeNode: Code, tree: Root): 'canon_fact_records' | 'change_log' | 'other'`, `narrowToCanonFactRecord(raw: unknown): CanonFactRecord | null`, `narrowToChangeLogEntry(raw: unknown): ChangeLogEntry | null`.

Algorithm:

1. Walk the mdast tree for every `code` node with `lang === 'yaml'`.
2. For each block: determine its parent section via `findContainingSection`. Blocks under `canon_fact_records` route to CF; blocks under `change_log` route to CH; everything else is parsed as generic YAML and emitted as a node with `node_type='section'` (the `subsection` containing it is resolved via the Pass 1 heading-path stack — delegated to the common node-synthesizer in SPEC-01-006).
3. Call `yaml.parse(block.value)`. On thrown error, emit a `ValidationResultRow` with `validator_name='yaml_parse_integrity'`, `severity='warn'`, `code='yaml_syntax_error'`, `message=<error message>`, populated `file_path` + `line_range_*`; and emit a partial `NodeRow` with `node_type` inferred from the section, `node_id=<synthetic id based on line range>` if no id can be extracted, `body=<raw YAML text>`, `content_hash=contentHashForProse(body)` (fall back to prose hash when YAML is unparseable).
4. On successful parse: run `narrowToCanonFactRecord` / `narrowToChangeLogEntry`. If the narrow fails (missing required field), emit a `yaml_parse_integrity` warn row naming the missing field; still emit the NodeRow using the parsed-but-shape-invalid object and `contentHashForYaml(parsed)`.
5. On successful narrow: use `parsed.id` (or `parsed.change_id`) as `node_id`; `body = <raw YAML text>` (NOT the re-serialized form — we preserve author whitespace in the node body while hashing with `contentHashForYaml`).

## Out of Scope

- Edge extraction from YAML fields (`source_basis.derived_from`, `modification_history[].change_id`, etc.) — covered by SPEC-01-006 Pass 3.

## Acceptance Criteria

### Tests That Must Pass

1. Extracting from `fixture-yaml-wellformed.md` produces one CF node + one CH node with correct `node_type`, `node_id`, and `content_hash` values.
2. Malformed fixture produces a `warn`-severity `yaml_parse_integrity` row + a partial node with raw body.
3. Missing-field fixture produces `missing_required_field:statement` warn + a node with `node_type='canon_fact_record'`.
4. Unknown-field fixture parses successfully with `exception_governance` accessible via `(record as any).exception_governance`.
5. CF and CH blocks with identical YAML content but different section ancestors produce different `node_type` values.

### Invariants

1. No parse path throws an uncaught exception that aborts the build — every failure mode routes through `validation_results` (Rule-6 / Rule-5 preservation: failures are observed, not silenced).
2. `body` of a YAML-backed node is the raw author text (preserves attribution register); the canonical hash is computed from `contentHashForYaml(parsed)` (independent of author whitespace).
3. No field name in the narrow-check helpers diverges from FOUNDATIONS.md §Canon Fact Record Schema — enforced by a per-field grep in the test harness.
