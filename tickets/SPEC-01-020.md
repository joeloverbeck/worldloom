# SPEC-01-020: Retune `named_entity` extraction against structural and workflow-label noise

**Status**: PENDING
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — `tools/world-index` heuristic entity extraction, source-node selection, and integration coverage for the live `animalia` corpus.
**Deps**: `archive/tickets/SPEC-01-018.md`, `archive/tickets/SPEC-01-019.md`, `docs/FOUNDATIONS.md`, `archive/specs/SPEC-01-world-index.md`

## Problem

After a fresh rebuild with the current `tools/world-index` code (`node tools/world-index/dist/src/cli.js build animalia` on 2026-04-22), the production `animalia` index is structurally clean but still semantically unreliable on the `named_entity` surface. The rebuilt DB contains `487` `named_entity` rows, `481` of them `Kind: unknown`, with obvious structural/workflow-label false positives such as `Mystery Reserve` (`159` mentions), `Continuity Archivist` (`77`), `Mystery Curator` (`63`), `Change Log Entry` (`25`), `Primary Difference` (`25`), `Natural Story Engines` (`22`), `Required Updates` (`12`), and `No Silent Retcons` (`5`).

Those rows are not stale-db residue; they reproduce after rebuild. They are being persisted as first-class entities and linked from `entity_mentions`, which weakens the machine-facing semantic layer that downstream retrieval and spec work rely on.

## Assumption Reassessment (2026-04-22)

1. `tools/world-index/src/commands/shared.ts` still defines `ENTITY_SOURCE_NODE_TYPES` broadly enough to feed heuristic entity extraction from `adjudication_record`, `audit_record`, `section`, `subsection`, and `bullet_cluster` nodes in addition to world-facing record nodes.
2. Fresh live audit after rebuild of `worlds/animalia/_index/world.db` shows the remaining problem is not structural integrity: `validation_results = 0`, unresolved edges = `0`, `verify animalia` returns cleanly, and there are no dangling edge or mention references.
3. Shared boundary under audit: `tools/world-index/src/parse/entities.ts`, `tools/world-index/src/parse/stoplist.ts`, `ENTITY_SOURCE_NODE_TYPES` in `src/commands/shared.ts`, persisted `named_entity` / `entity_mentions` rows in SQLite, and downstream retrieval/audit consumers that query those rows directly.
4. FOUNDATIONS Core Principle says canon facts must reinforce a constrained world model. Promoting template headings (`Primary Difference`, `Natural Story Engines`), repo/workflow phrases (`No Silent Retcons`, `Required Updates`, `Change Log Entry`), and process-role labels (`Continuity Archivist`, `Mystery Curator`) into world entities weakens that contract rather than strengthening it.
5. Live DB evidence shows the dominant noise source is adjudication content rather than world prose: `entity_mentions` grouped by source node type yields `adjudication_record = 1447` mention rows / `345` distinct entities, far above the next source type (`section = 337`).
6. The bad entities are not edge-case one-offs. Grouped live queries on the rebuilt DB show:
   - `Mystery Reserve` appears across `world-root`, `adjudications`, and `proposals`
   - `Change Log Entry` appears across `adjudications` and `audits`
   - `Required Updates` appears from `adjudications` only
   - `No Silent Retcons` appears across `adjudications` and `audits`
   - `Continuity Archivist` / `Mystery Curator` are sourced primarily from adjudication workflow prose
7. Mismatch + correction: `archive/tickets/SPEC-01-018.md` already removed a first wave of operational-label noise (`Adds Mystery Reserve`, `All Phase`, `Age Details`, `Action Report`, `Access Path`), but the rebuilt live corpus proves the remaining false positives are broader than a small stoplist-only seam. This follow-up should reassess both source-node scope and filtering heuristics.

## Architecture Check

1. The clean fix is to make entity extraction more selective about which node classes and phrase classes are eligible, rather than continuing to store obvious workflow/template labels and expecting downstream consumers to subtract them later.
2. No backwards-compatibility aliasing or duplicate semantic surfaces should be introduced. The end state is a cleaner `named_entity` graph, not a parallel "filtered entities" layer.

## Verification Layers

1. Structural/workflow labels no longer persist as `named_entity` rows after a fresh `animalia` rebuild. -> readonly DB query against banned examples
2. Legitimate retained entities from world content still persist. -> targeted DB query for known positives
3. The heuristic source surface no longer lets adjudication workflow/report scaffolding dominate entity output. -> DB query by source node type / file family
4. The rebuilt DB remains structurally clean (`verify`, unresolved edges, validation rows). -> command + readonly DB queries

## What to Change

### 1. Reassess eligible source nodes for heuristic entity extraction

Audit whether `adjudication_record`, `audit_record`, broad container nodes, or some subset of them should contribute heuristic entities at all. If they remain eligible, tighten extraction so workflow/report scaffolding inside those nodes does not emit persisted entities.

### 2. Extend filtering beyond the first-wave stoplist

Retune `tools/world-index/src/parse/entities.ts` and `src/parse/stoplist.ts` so template headings, report section labels, repo-rule phrases, and workflow-role labels stop becoming `named_entity` rows in the live corpus.

### 3. Add live regression coverage at the `animalia` seam

Encode the rebuilt-animalia false positives directly in integration coverage so the semantic layer cannot drift back toward workflow-label pollution without a failing test.
