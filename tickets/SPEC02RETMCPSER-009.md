# SPEC02RETMCPSER-009: Approval token signing + .secret lifecycle

**Status**: PENDING
**Priority**: MEDIUM
**Effort**: Small
**Engine Changes**: Yes — introduces `tools/world-mcp/src/approval/token.ts` and the `.secret` lifecycle (gitignored).
**Deps**: SPEC02RETMCPSER-001

## Problem

`mcp__worldloom__submit_patch_plan` (tool 9) requires an `approval_token` that binds user-consent-at-HARD-GATE to the exact bytes of the patch plan. This ticket lands the MCP-side signing half: HMAC-SHA256 over `{plan_id, world_slug, patch_hashes[], issued_at, expires_at}`, base64-encoded, using a per-install secret file at `tools/world-mcp/.secret` (gitignored, generated on first run). The verification half (signature check + `approval_tokens_consumed` single-use enforcement) lives in SPEC-03 engine, which reads the same secret file. Secret-sharing between MCP and engine is the only cross-package filesystem dependency in the machine-facing layer.

## Assumption Reassessment (2026-04-23)

1. `tools/world-mcp/src/approval/` directory exists as `.gitkeep` placeholder from -001. `.gitignore` already includes `tools/world-mcp/.secret` per `specs/IMPLEMENTATION-ORDER.md:22` (Phase 0 completed 2026-04-22); no `.gitignore` edits are needed.
2. `specs/SPEC-02-retrieval-mcp-server.md` §Approval token (lines 230–243) is the authoritative source for the signing payload, base64 wrapping, single-use-plus-expiry semantics, and the `.secret` location. `specs/SPEC-03-patch-engine.md:192` confirms the engine verifies against the same secret (cross-package dependency is deliberate per this spec's Improvement 1 finding in the reassessment).
3. Cross-package boundary under audit: `tools/world-mcp/.secret` is read by BOTH world-mcp (signing) and patch-engine (verifying). The secret is generated by whichever process runs first; both must tolerate the file existing with 0600 permissions.
5. HARD-GATE semantics: the approval token is the mechanism that ties user HARD-GATE approval to specific plan bytes. A skill that submits a *different* patch plan than the one the user approved will fail verification because `patch_hashes[]` will not match. This ticket must produce tokens that are byte-tamper-detectable.

## Architecture Check

1. HMAC-SHA256 via `node:crypto` is cleaner than rolling custom crypto; pinned by the standard library, no external dep added.
2. A secret file at package root (not an env var) is cleaner for per-install binding — the file survives shell restarts, integrates naturally with `.gitignore`, and scopes the secret to one worktree. An env-var approach would require the user to re-set it after every shell restart.
3. No backwards-compatibility shims.

## Verification Layers

1. Token round-trip → unit test signs a payload, parses the base64 string, recomputes the HMAC, verifies match.
2. Tamper detection → unit test flips one byte of the payload, asserts HMAC mismatches (the engine-side verifier will reject; this ticket only needs to prove the signature is computed over the exact payload bytes).
3. Secret lifecycle → unit test on an empty package directory: first call to the signer creates `.secret` with 0600 permissions; second call reuses the file; third call (with manual secret deletion) regenerates with different entropy.
4. Expiry enforcement → unit test signs with `expires_at` 5 minutes from now, asserts the token's embedded `expires_at` is future; a second test signs with `expires_at` in the past, asserts signer still produces the token (expiry enforcement is engine-side; signer does not refuse expired tokens) but logs a warning.

## What to Change

### 1. `tools/world-mcp/src/approval/token.ts`

Export:

```typescript
export interface ApprovalTokenPayload {
  plan_id: string;
  world_slug: string;
  patch_hashes: string[];
  issued_at: string;   // ISO timestamp
  expires_at: string;  // ISO timestamp; signer defaults to issued_at + 5min
}

export function readOrCreateSecret(): Buffer;       // reads tools/world-mcp/.secret or creates it (32 random bytes)
export function signToken(payload: ApprovalTokenPayload, secret: Buffer): string; // returns base64(payload || signature)
export function parseToken(token: string): { payload: ApprovalTokenPayload; signature: Buffer };
```

`signToken` concatenates `JSON.stringify(payload)` + `'.'` + HMAC-SHA256 signature, then base64-encodes the whole string. Matches spec §Approval token lines 232–238.

### 2. Secret file lifecycle

- On first `readOrCreateSecret()` call: `fs.writeFileSync(secretPath, crypto.randomBytes(32), {mode: 0o600})`.
- On subsequent calls: `fs.readFileSync(secretPath)`.
- No rotation mechanism in Phase 1 (spec §Risks line 325 flags rotation as a Phase 2+ concern).

### 3. Tests

- `tests/approval/token-lifecycle.test.ts` — sign + parse round-trip, tamper detection, secret creation/reuse/regenerate.

## Acceptance Criteria

### Tests That Must Pass

1. `cd tools/world-mcp && npm test` — `token-lifecycle.test.ts` passes.
2. First run in a fresh directory creates `.secret` with mode `0600`; verified via `fs.statSync(secretPath).mode & 0o777 === 0o600`.
3. Signed token is base64-decodable and the decoded payload parses to the exact input object.
4. Tamper test: modifying one byte of the base64 token before parsing breaks HMAC verification when recomputed.

### Invariants

1. `.secret` is always created with mode 0600; permissive modes are a bug.
2. `.secret` is never logged, printed, or returned from any exported function.
3. Signed tokens always include `issued_at` + `expires_at` ISO timestamps.
4. `expires_at` defaults to `issued_at + 5 minutes` when the caller does not override it (spec §Approval token line 242).
