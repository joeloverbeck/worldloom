# Skill Audit

Analyze a skill file against the work done in the current Claude Code session to determine whether the skill has issues, could be improved, or needs new features. Report only — never modify the target skill.

## Invocation

```
/skill-audit <path-to-skill-directory>
```

Example: `/skill-audit .claude/skills/brainstorm`

The argument is the skill directory path. The framework automatically resolves `SKILL.md` within it.

## Checklist

1. **Read the target skill** — Read the SKILL.md file at the provided path. Parse its name, description, and full content. The Read requirement is satisfied by in-context content from any prior operation in the current session — direct Read tool invocations, skill-invocation-embedded content (when the target skill was invoked earlier in the session and its full body was injected via that invocation), or system-reminder inclusions — parallel to Step 2's rule for alignment documents. Re-read only when content is not in context (e.g., after compaction or context rotation) or when the in-context version is suspected stale. If the target skill was modified during the current session (via user-directed follow-up implementation from a prior audit, or any other in-session edit), re-read is mandatory — parallel to Step 3's self-audit "re-read the file at Step 1 regardless of prior reads" rule for the same reason. For follow-up implementation (Guardrails §Follow-up implementation) that will Edit the target, a fresh Read at that phase is recommended regardless of audit-phase Read status, to ensure exact old_string match. If the exact path does not resolve, glob for `<path>*/SKILL.md` (appending wildcard + `/SKILL.md`). If that also fails, try `<path>**/SKILL.md`. If exactly one match is found, use it and note the correction. If zero or multiple matches, stop and report the error.

   **Auxiliary investigation (permitted, with required announcement).** Beyond reading the target SKILL.md, you may additionally: (a) list contents of the target skill's directory and of any directly-referenced sibling directories; (b) diff files against named reference sources when the target's text claims a reference-source relationship (e.g., "copy from skill-creator's own templates" — check whether those templates actually ship and whether sibling skills have drifted from them); (c) glob for shared-surface files (`archive/*/`, `brainstorming/*.md`, `worlds/*/`) when a potential finding hinges on their presence or naming. Each auxiliary investigation must support a specific hypothesis about the target skill's behavior — do not probe speculatively.

   **Recommended at audit start**: when the target skill ships a `references/` or `templates/` subdirectory, list the directory contents as the audit's first auxiliary action. This surfaces the full file structure so per-finding Suggestion fields can cite exact file paths (per Step 7's Suggestion specificity sub-rule) and the implementation phase doesn't have to discover file structure mid-flow via cascade scan. The listing still requires a hypothesis-first announcement under the rule above (e.g., `Investigating which references/ files this skill ships: listing the skill directory.`). Skip this recommendation only when the target skill is structurally a single SKILL.md with no subdirectories.

   **Announcement rule** (applies to any tool call beyond the target Read): announce hypothesis-first as user-facing text IMMEDIATELY BEFORE the tool call. Format: `Investigating <hypothesis>: <grep/read/diff/list> <target> to verify.` Examples:

   - `Investigating whether templates/ actually ship in this skill: listing the skill directory.` (setup-feeling action — routine directory listing at audit start still counts as auxiliary investigation and needs announcement)
   - `Investigating whether sibling meta-tooling skills reference changed terminology: grepping .claude/skills/*/SKILL.md for the affected tokens.` (substantive follow-up investigation mid-audit)

   Routine setup actions (listing the target directory, loading sibling files referenced from SKILL.md) require announcement just as substantive investigations do — the rule draws no line between "interesting" and "boring" auxiliary calls. Tool `description` fields do NOT satisfy this rule — they don't render in the transcript where reproducibility needs them. Hypothesis-first phrasing makes the audit trail reproducible; a reader can reconstruct what was being tested vs. which tool answered it.

   **Batched tool-call announcements.** When multiple tool calls are batched in a single assistant turn (parallel tool-call batching per the Edit-ordering rule elsewhere in this skill), ONE announcement covering the batch is acceptable if all calls investigate the same hypothesis. Example: `Investigating whether X and Y both ship: listing both directories.` covers a 2-call batch with one announcement when the hypothesis is joint. If calls investigate DISTINCT hypotheses, each needs its own announcement in the same pre-batch prose: `Investigating whether X ships: listing the directory. Separately investigating the Y convention: grepping siblings.` — two hypotheses, two announcements, one batch. The batch-one-announcement shortcut is for shared-hypothesis batching only; it is not a general dispensation to elide announcements.

   **Cascade-scan batching across multiple findings.** Cascade scanning produces TWO distinct artifacts emitted at different times: **(a)** a pre-grep per-finding hypothesis announcement (e.g., `Investigating Improvement 1's cascade targets: grepping for <term>`) — emitted as user-facing text immediately before the parallel grep batch; and **(b)** a post-grep per-finding outcome line (per the §Quick-reference cascade-scan outcome by shape table in §Cascade edits — `Finding N: <count> hits found, all accounted for as primary sites...`) — emitted as the per-finding cascade-scan visibility line before the post-implementation summary table. Both are required for full audit-trail reproducibility; emitting only the outcome line drops the hypothesis context that lets a future reader reconstruct what was being tested, and emitting only the announcement drops the verifiable result a future reader needs to confirm the scan ran. The common implementation-phase case — cascade scans for N findings batched into one parallel tool call — constitutes **distinct hypotheses (one per finding), not one shared hypothesis**. Each finding's cascade scan investigates "are there cascade targets for THIS finding's primary edit?", which is a per-finding question even when the greps run in parallel for efficiency. Announce each finding's scan separately in the pre-batch prose, naming the finding and its scan concern: `Investigating Improvement 1's cascade targets: grepping for <term-1>. Separately investigating Improvement 3's cascade targets: grepping for <term-2>. Separately investigating Feature 1's cascade targets: grepping for <term-3>.` Collapsing these into a single umbrella announcement like "Investigating cascade-scan targets across all findings" breaks the per-finding reconcilability that the post-implementation summary's `N.cascade` keying depends on — a reader of the summary must be able to trace each cascade row back to its planning-phase scan, and a shared umbrella announcement forces guess-work at the mapping. When the scan outcomes diverge per finding (some produce cascades, some produce complementary hits, some produce no hits — per the Quick-reference cascade-scan outcome table in §Cascade edits), per-finding announcements before the batch let the per-finding outcome lines after the batch land on an already-established per-finding frame. **Prefer parallel tool-call batching** (one Bash invocation per finding's grep, sent as multiple tool blocks in a single assistant turn) over single-Bash-command batching (greps stitched with `&&` and `echo` section headers). The parallel-tool-call form bounds each finding's output independently; the single-Bash form risks total-output truncation when combined hits are large, forcing re-runs that pollute the audit trail and break the per-finding announcement-to-result reconcilability. Single-Bash stitching may look efficient (one tool call instead of N) but trades that for an unbounded combined-output budget — the truncation failure mode is non-obvious until it fires, and the recovery (separate re-run for the truncated finding) leaves the audit trail with a delayed result that the original announcement-to-result pairing no longer covers cleanly.
2. **Read alignment documents** — Read `docs/FOUNDATIONS.md` — skip only if read earlier in this session (fully or via partial reads that cumulatively covered the document), not from memory or training knowledge. "Read earlier in this session" includes FOUNDATIONS content loaded into this session's context by any prior operation — direct Read tool invocations, skill-invocation-embedded content (when a prior skill's Phase read the file as part of its own procedure), or system-reminder inclusions all satisfy this criterion. Re-read only when the content is not in context (e.g., after a compaction or context rotation — or when the earlier read covered only unrelated sections and this audit's topic requires a different section). If the file exceeds the Read tool's token limit, read the first 200 lines (preamble + principle listing) using offset/limit, or read relevant sections targeted to the audit topic. Multiple partial reads that cumulatively cover the full document satisfy this requirement. If `CLAUDE.md` exists at the repository root, also read it (if `CLAUDE.md` is absent — e.g., a subdirectory or worktree without one — treat its absence as normal and skip the CLAUDE.md alignment check in that case). For meta-tooling skill targets (e.g., brainstorm, skill-creator, skill-audit, and similar process/tooling skills), this FOUNDATIONS.md read may be skipped — alignment will be N/A per Step 4.
3. **Session reflection** — Review the current conversation context to identify the items below. If the target skill is skill-audit itself (self-audit), use session evidence from any prior audit invocation(s) in this session. Session evidence spans both the audit phase (Steps 1-7) and any user-directed follow-up implementation flow (per Guardrails §Follow-up implementation) — friction in either phase is in-scope for a self-audit. The self-audit invocation provides no independent session evidence beyond confirming the skill's flow works. If no prior audit invocation exists in this session, report "No session evidence available — self-audit with no prior invocations produces no findings beyond confirming the skill's flow parses correctly." and skip steps 3-6. If skill-audit itself was modified during the current session (via user-directed follow-up implementation on an earlier audit, or any other in-session edit), base the self-audit on the CURRENT skill content — re-read the file at Step 1 regardless of prior reads. Session evidence then spans both pre- and post-modification behavior; explicitly mark any prior-session observation that an intervening edit already resolved as "resolved — no action needed" rather than re-raising it as a fresh finding.
   - Moments where the skill's instructions were unclear or ambiguous
   - Steps that were skipped, reordered, or worked around
   - Behaviors the skill didn't anticipate (edge cases, unexpected inputs)
   - Places where Claude had to improvise because the skill didn't provide guidance
   - Outcomes that diverged from what the skill intended
   - Steps that were not exercised this session (mark as "not exercised" — do not speculate about issues)
4. **Cross-check alignment** — For each finding from step 3, check whether the skill contradicts or fails to implement:
   - Principles from `docs/FOUNDATIONS.md` (reference by foundation number)
   - Conventions from `CLAUDE.md` (reference by section name) — skip this bullet entirely if `CLAUDE.md` is absent from the repo root
   - For meta/tooling skills that do not touch canon/world-model design (e.g., brainstorm, skill-creator, skill-audit), note "N/A — meta-tooling skill, FOUNDATIONS principles do not apply" and move on. Reserve detailed alignment analysis for skills that govern canon-pipeline proposals, specs, or tickets.
5. **Classify findings** — Categorize each finding into one of three buckets:
   - **Issue**: Something broken, misleading, or contradictory in the skill
   - **Improvement**: A refinement to existing behavior that would make the skill more effective
   - **Feature**: A new capability that aligns with the skill's stated intent but is currently missing
6. **Severity-tag each finding** — CRITICAL / HIGH / MEDIUM / LOW. Use this rubric:
   - **CRITICAL**: Skill produces wrong output, corrupts state, or violates a FOUNDATIONS principle. Must fix before the skill is used again.
   - **HIGH**: Missing guardrail or instruction that has already caused rework or wrong output in this session, or a plausibly near-term failure mode on the next use.
   - **MEDIUM**: Friction that cost non-trivial improvisation or required non-obvious judgment to work around. The skill still produced the right outcome, but the path was not smooth. MEDIUM examples include: (a) skill's literal rule contradicts established precedent, forcing you to invoke precedent to satisfy validation (following the skill literally would produce incorrect output); (b) skill's documented enum or convention is missing values the operational world uses, forcing you to choose between skill compliance and correct output; (c) a skill step's format constraint doesn't fit the delivery shape without restructuring.
   - **LOW**: Wording refinement, coverage gap, or polish. Did not block progress and a competent operator could work past it without guidance. LOW examples include: wording that is slightly confusing but parseable on second read; an example that could be richer; a cross-reference that is valid but indirect; a convention drift between paired documentation that doesn't affect output correctness.
7. **Present the report** — Output the structured report using the template below.

## Report Template

Output this structure to the conversation (do not write to a file):

```markdown

**Skill path**: <path>
**Session date**: YYYY-MM-DD

## Alignment Check

- **FOUNDATIONS.md**: <aligned / N violations found / N/A — meta-tooling skill>
- **CLAUDE.md**: <aligned / N deviations found / skipped — not present>
[If violations: bullet list with specific foundation # or CLAUDE.md section + what conflicts]

## Issues

[If none: "No issues identified."]

## Cross-Skill Scan

**Default**: OMIT this section when no finding affects shared surfaces with sibling skills. [When included, use one of four forms depending on what actually happened during the audit phase: (1) "Scanned: <sibling-skill-list> — no inconsistencies." when a scan was executed and found no drift; (2) "Scanned: <list> — adjusted X in Y." when a scan was executed and surfaced an inconsistency to fix; (3) "Cascade planned for implementation: <sibling-file> per <structural-identity claim or shared-surface trigger>." when a finding identifies a required cross-skill cascade (e.g., a paired template with a 'structurally identical to <sibling>/...' claim) that will be applied at implementation time rather than during the audit — this form fits non-meta-tooling targets where the sibling-scan operation doesn't apply but the finding nonetheless implicates a shared surface; (4) "Scanned: <list> via <method> — cascade-target candidates found; per-sibling investigation narrowed scope (<reason per excluded sibling>); no sibling cascade planned." when a sibling scan ran during the audit phase's Auxiliary investigation window, found initial cascade-target candidates, and scope-narrowed to zero cascades (uncommon at audit phase; more often the Mid-implementation cascade scope-narrowing sub-rule fires at implementation phase — see Guardrails §Cross-skill consistency — but when scope-narrowing happens at audit phase the same reasoning applies: categorical-different surface shape, absent parallel section, or sibling-coherence-breaking content). **Suggestion specificity**: When a finding's primary edit could land in either `SKILL.md` or a `references/` / `templates/` file, cite the exact file path in the Suggestion field (e.g., `Add to references/X.md §Section` rather than just `§Section`). When the section name is unique across the skill's files, the bare `§Section` form is acceptable. The audit-phase directory listing recommended in Step 1's auxiliary-investigation block surfaces which case applies — running it at audit start makes this contract verifiable rather than ambiguous, and prevents the implementation phase from re-keying primary/cascade rows when a section name turns out to live in a deeper reference file the audit didn't see.
```

Double-check severity counts against findings before presenting. If a correction is needed after presenting, strike the incorrect line and restate.

**All findings implement by default**. "Implement all", "implement recommended", "implement suggestions", and any similar inclusive phrasing (any "implement" request that does not name specific findings by number) are synonymous: all apply every numbered Issue/Improvement/Feature in the report. The baseline assumption is that any finding worth numbering and presenting is worth implementing.

If a finding is worth surfacing but NOT worth auto-applying, tag it explicitly on the title line:
- ` — skip` — the auditor considered applying this and decided against it (e.g., two valid directions with no clear winner, user preference needed before choosing)
- ` — informational` — context the user should know, but it does not translate to a concrete code change
- ` — no change needed` — append to the Suggestion line (not the title) when the finding's conclusion is that the current behavior is already sufficient

Explicitly-tagged findings are excluded from "implement all" / "implement recommended" scope. Everything else is applied.

Example: `1. **[LOW]** Tighten batching threshold` (applied by default); `2. **[LOW]** Alternate naming convention — informational` (surfaced for awareness, not applied).

Rationale for this default: the previous "tag to include" pattern created inconsistency — the same finding might be tagged recommended in one session and untagged in another, depending on auditor judgment with no rubric. Flipping the default aligns the skill's behavior with user expectation ("implement recommended" = "implement all") and removes the tagging-judgment burden unless the auditor has a specific reason to hold back.

## Guardrails

- **Report only** — Never modify the target skill file. Output the report to the conversation only.
- **No false positives** — If a step in the skill wasn't exercised during the session, note "not exercised this session" rather than speculating about potential issues.
- **FOUNDATIONS alignment is mandatory** — Any suggestion that would violate a principle in `docs/FOUNDATIONS.md` must be flagged and rejected, even if it would otherwise be an improvement.
- **Scope discipline** — Do not propose expanding the skill's scope beyond its stated intent. The audit evaluates the skill as written, not what it could become.
- **Session evidence required** — Every Issue and Improvement must cite specific session evidence (what happened, what was expected). Findings based purely on hypothetical scenarios belong in Features, not Issues.
- **Follow-up implementation** — After the report is presented, the user may request implementation of specific suggestions. At that point, edit the target skill file directly — the "report only" guardrail applies only to the audit phase, not to user-directed follow-up.

  **Common path (checklist)**:
  1. Check if any skill file covered by findings (SKILL.md plus any `references/*.md` or `templates/*.md` files cited in the audit's per-finding Suggestion or Skill gap fields) has changed since the audit report → if yes, run **Re-evaluation** (below).
  2. Read user request: partial ("implement 1, 3") vs. full ("implement all" / "implement recommended" — synonymous).
  3. Apply edits in document order (top → bottom); scan each for **cascades** before applying.
  4. Re-read all edited files; run the **post-edit verification** pass (5 checks for non-meta-tooling targets; 6 checks when the target is a meta-tooling skill).
  5. Present the **post-implementation summary**: per-batch cascade-scan line FIRST — confirms the scan ran even when it found nothing; see the `Per-finding cascade scan visibility` sub-rule below — THEN the table (Finding | Severity | Status). Placing the cascade-scan line AFTER the table breaks the audit-trail ordering: readers should see scan discipline confirmed before they interpret table rows.
  6. If a stated structural-identity claim in the target forces a sibling-file mirror, apply the **cross-skill cascade** (keyed `N.cascade` in the summary). AND, separately, if the target is meta-tooling and any edit touched shared terminology or conventions, run the **cross-skill sibling scan** (with a named scan method) and document both in the summary. The two operations are independent — see Guardrails §Cross-skill consistency for the distinction. **When the sibling scan is unnecessary** — i.e., the target is NOT meta-tooling AND no structural-identity cascade applies AND the edits don't touch shared surfaces — record a single canonical line in the post-implementation summary of the form `Cross-skill sibling scan: Scan not performed — <reason>` (e.g., `Scan not performed — target is canon-mutating/non-meta-tooling; consumer-side dependencies already verified at audit phase`). Use the literal `Scan not performed — <reason>` phrasing; paraphrased variants (e.g., "the sibling-scan operation does not apply") break grep-searchability for future audits tracing scan discipline across sessions. **When the sibling scan IS run and scope-narrows to zero cascades** — i.e., per-sibling investigation during implementation found cascade-target candidates but the Mid-implementation cascade scope-narrowing sub-rule (see Guardrails §Cross-skill consistency) narrowed scope per categorical-different-shape / absent-parallel / sibling-coherence-breaking reasons — record the canonical line of the form `Cross-skill sibling scan: Scanned: <sibling-list> via <method> — scope narrowed per Mid-implementation cascade scope-narrowing rule (<reason per excluded sibling>); no sibling cascade applied.` This form preserves the audit trail that a scan ran AND reached a deliberate zero-cascade outcome, distinguishing from `Scan not performed — <reason>` (which says no scan ran at all) and from the three positive forms (which describe different outcomes). Use the literal `scope narrowed per Mid-implementation cascade scope-narrowing rule` phrasing for grep-searchability parallel to the `Scan not performed` discipline above. **When the target IS meta-tooling but no edit touched shared surfaces** — i.e., the target is a meta-tooling skill (brainstorm, skill-creator, skill-audit, skill-consolidate, etc.) AND no edit touched the surfaces enumerated in §Concrete shared-surface triggers AND no structural-identity cascade applies — the scan is OPTIONAL. If executed defensively, use the standard positive form (`Cross-skill sibling scan: Scanned: <list> via <method> — no inconsistencies.`). If skipped, record the canonical line `Cross-skill sibling scan: meta-tooling target with no shared-surface edits in this implementation; defensive scan skipped.` for grep-searchability. This form fills the matrix gap between `Scan not performed — <reason>` (reserved for the non-meta-tooling case per the first form above) and the positive `Scanned: ... — no inconsistencies` form (which asserts an executed scan); without an explicit canonical phrase, a less careful operator could skip silently and leave no audit-trail indication that the scan was even considered. Use the literal `meta-tooling target with no shared-surface edits in this implementation; defensive scan skipped` phrasing for grep-searchability parallel to the prior canonical-form disciplines above.

  The sub-rules below are the full specification — use this checklist to orient, then consult the sub-rules (Re-evaluation, Partial implementation, Edit ordering, Cascade edits, Post-implementation summary, Post-edit verification) for edge cases and worked examples.

  **Re-evaluation**: If any skill file covered by findings (SKILL.md plus any `references/*.md` or `templates/*.md` files cited in the audit's per-finding Suggestion or Skill gap fields) changed between the audit report and any follow-up-implementation batch — whether from an external edit (a different session or user action outside the audit flow), a previous partial-implementation batch in this same session, or a user-directed edit to the target between report and follow-up — re-evaluate each finding against the current state before applying edits. **How to check**: before applying any edits in the implementation batch, re-Read the file regions covered by each finding's primary edit; if the audit's quoted old-string text is no longer at the cited location, the file has changed and Re-evaluation is required. The per-finding cascade scan during edit application is NOT a substitute for this region re-Read; cascades may serve as a backstop when a finding's primary-edit text is terminologically related to other content, but the primary detection mechanism is the explicit pre-edit region re-Read covering each finding's primary-edit location. Discard obsolete findings, adapt shifted ones, and renumber survivors. Announce the re-evaluation explicitly before applying any edits in the new batch: "Re-evaluating N remaining findings against the modified target file..." followed by a one-line outcome per finding (`still-valid` / `target-shifted — adapted` / `now-moot — discarded`). When no edits have touched the target files between the report and the current batch, the full per-finding announcement may be replaced by a single confirmation line (e.g., "No target edits since audit; proceeding without re-evaluation.") or omitted entirely. The full per-finding announcement is required only when a target file has actually changed. **Discarded-row reason field**: discarded rows in the post-implementation status table use the canonical status `now-moot — discarded` followed by a brief reason field naming what changed (e.g., `now-moot — discarded; references/X.md was rewritten between audit and implementation to assume Y had landed`). The reason field is required, parallel to the cascade-row reason convention — both serve audit-trail reproducibility.

  **Partial implementation**: If the user requests specific findings (e.g., "implement 1 and 3"), check whether skipped findings depend on implemented ones. If so, note the dependency and ask whether to include the dependent finding. If the user requests implementation without naming specific findings (e.g., "implement all", "implement recommended", "implement suggestions", or any similar inclusive phrasing — all synonymous under the default semantics), skip dependency checking and apply every finding in document order except those explicitly tagged `— skip`, `— informational`, or `— no change needed` in the audit report.

  **Finding identifier convention**: Findings are identified by section-prefixed numbering from the Report Template (`Issue 1`, `Issue 2`, ...; `Improvement 1`, `Improvement 2`, ...; `Feature 1`, ...). Each section has its own 1-based counter. When the user requests "implement 1 and 3", default interpretation is "Issue 1 and Issue 3"; if the audit has fewer Issues than the requested number, confirm the section before proceeding ("did you mean Improvement 1 and Improvement 3?"). For the post-implementation summary table, prefer section-prefixed row keys (`Issue 1`, `Improvement 1`, `Feature 1`) so the reader can reconcile row counts to the audit's per-section counts at a glance; flat-continued numbering (1–N across all sections) fuses two schemes and is discouraged.

  **Edit ordering**: Apply edits in document order (top to bottom) to minimize line-number shifts invalidating later edits. **Parallel Edit tool-call batching** (multiple Edit blocks sent in a single tool call) is equivalent to sequential document-order application when each `old_string` is unique and non-overlapping — Edit matches on exact string content, not line position, so shifts from one edit cannot invalidate another's match. Prefer parallel batching when old_strings are unique and non-overlapping; fall back to sequential only when an edit's `new_string` would be re-matched by a later edit's `old_string` (creating ambiguity) or when the edits touch overlapping text regions. This tool-call-granularity optimization is distinct from the sub-letter `N.a` / `N.b` "parallel edits" pattern documented in the post-implementation summary section — that pattern is about semantic co-equality of edits serving one finding; the optimization here is about tool-call execution shape. For multi-file targets (skills with `references/` or `templates/` sub-directories where findings distribute across multiple files), batch all edits within one file before moving to the next — do NOT interleave by finding number across files. File-batch order may follow the target's own load order (as listed in its SKILL.md Procedure steps), Phase/Step order, or alphabetical order of filenames — any order that avoids per-file re-reads between findings and keeps each file's cascade scan localized. Within each file, apply edits top-to-bottom per the primary rule. If applying an earlier finding renders a later finding moot (e.g., the target text no longer exists), skip the moot finding and note it in the post-edit verification as "superseded by finding N." If an Edit call fails with "String to replace not found in file," the most common causes are (a) a prior edit in the same batch shifted the target text, or (b) the `old_string` was constructed from memory rather than from a fresh Read and doesn't match the baseline exactly. In either case, use Grep or a targeted Read to find the current text, then retry with the corrected `old_string`. If an edit inserts a new numbered step, renumber all subsequent steps and verify that the output summary or other sections referencing step numbers are updated accordingly. When an edit inserts or removes a numbered item, scan preceding edits in the same batch for cross-references to the shifted numbers and fix them in the same edit or immediately after.

  **Restructure-encompassing findings**: When a Feature-level finding prescribes an architectural restructure (new file creation, extraction to `references/`, file split, consolidation) that encompasses the surface area of other findings' primary edits, apply the Feature's restructure first and bake each encompassed finding's content into the new structure as it is being created. Status-key each encompassed finding by its normal number with its primary-edit location pointing to the new file path (e.g., `implemented — primary edit in references/phase-6-canon-safety-check.md`); key the Feature itself with a descriptive reason naming the restructure scope and the encompassed findings (see the dedicated Quick-Reference row and Worked example in the Post-implementation summary section). This is distinct from the meta-finding shape (which has no dedicated edit); the Feature HAS a dedicated architectural edit (the file creation itself), but that edit physically houses other findings' content. Without this pattern, strict document-order application forces the monolithic-edit-then-restructure double-work path — other findings' content lands in the monolithic SKILL.md first, then has to be moved into the new references/ files when the restructure runs, rewriting the same content twice. Applying the restructure first and baking in during file creation avoids the double-work.

  **Mid-implementation primary-edit scope-narrowing**: If per-instantiation investigation at implementation time reveals that a planned primary edit site has a categorically different shape than the audit suggestion implied (e.g., a suggested "mirror in all class-specific checks" would not make semantic sense for one class because that class's surface shape rules the proposed change out), it is acceptable to narrow the primary-edit scope. Announce the scope-narrowing explicitly in user-facing text before applying the remaining primaries, naming the excluded site and the categorical-different-shape reason: `Primary-edit scope decision: <excluded site> excluded because <reason>; applying primary at <remaining sites>.` Status-key the row(s) you DID apply per the normal `N.a` / `N.b` / ... convention with a sub-key count reflecting actual applied sites (not originally-implied sites). Do NOT silently skip divergent primary sites — the announcement preserves the audit trail that an audit-suggestion-to-implementation gap existed and was resolved with judgment rather than by oversight, and it lets a future reader reconstruct why the post-implementation summary has fewer primary rows than the audit's Suggestion field implied. This is distinct from Re-evaluation (which addresses target-file changes between audit and implementation) and from Partial implementation (which addresses user-specified finding subsets); primary-edit scope-narrowing is operator-directed at implementation time when the audit suggestion's site count was implicit or interpretable. Symmetric with §Mid-implementation cascade scope-narrowing in Guardrails §Cross-skill consistency — the same reasoning ("categorical-different surface shape, absent parallel section, sibling-coherence-breaking content" adapted to within-target instantiations) applies to primary edits as to cascades. Worked precedent: a skill-creator audit's Improvement 1 suggested adding a fourth scope label and "mirror in Step 6 canon-reading checks" — implementation-time investigation revealed canon-mutating skills don't write at root-level surfaces, so the fourth label can't apply to canon-mutating's Step 6 check; scope was narrowed to canon-reading + meta-tooling Step 6 sites (3 primaries — Step 4 gap-filler + Step 6 canon-reading + Step 6 meta-tooling — keyed `1.a` / `1.b` / `1.c`) with the canon-mutating site explicitly excluded and the reason announced before applying.

  **Cascade edits**: After planning the primary edit(s) for a finding, scan the rest of the file for related text that uses the same terminology, references the same concept, or would become inconsistent if only the finding's target text is changed. Apply cascade edits alongside the finding's primary edit. Cascade discovery may happen at two valid points: (a) per-finding during planning (preferred — matches the scan-after-planning flow above and keeps cascade + primary in the same batch step), or (b) during post-edit verification as a backstop when the per-finding scan missed something. When a cascade is discovered in post-edit verification, apply it immediately and note it in the post-implementation summary as usual. **Verification-backstop labeling**: when a cascade is caught by the verification backstop rather than planning-time scanning, prefix its reason field with `verification-backstop catch:` so the audit trail records where the per-finding scan missed. The backstop working is a success; the backstop being the only thing that caught the miss is a tuning opportunity for the planning-time scan discipline — the explicit label preserves both signals. No new outcome class is needed in the cascade-scan-outcome taxonomy; the label is a reason-field annotation on an existing `cascade` outcome. The verification backstop is not a replacement for per-finding scanning — it catches misses; it does not absolve the planning step. Note cascade edits in the post-implementation summary as "cascade from finding N."

  Cascades may be **intra-skill** (same skill's `SKILL.md` → its own `references/*.md` or `templates/*.md`) or **cross-skill** (target skill → sibling skill). Both use `N.cascade` keying in the post-implementation summary and follow the same scan-during-planning discipline above. The audit-phase Cross-Skill Scan section of the report template and the implementation-phase cross-skill sibling-scan operation (see §Cross-skill consistency) apply ONLY to cross-skill cascades — they do NOT fire for intra-skill cascades, which are handled by the standard cascade discipline alone. When deciding whether a cascade is intra-skill or cross-skill, check whether the cascade-target file lives under the same skill directory as the primary edit: same directory → intra-skill; different skill directory → cross-skill.

  **Per-finding cascade scan visibility**: When per-finding scanning produces no related text for any finding in the batch, report the absence as a single per-batch line BEFORE the post-implementation summary table (e.g., "Per-finding cascade scans: no cascades needed for any of N findings.") so the audit trail records that the scan happened. When some findings produced cascades and others did not, the cascade rows (`N.cascade`) cover the affirmative cases and the per-batch line need only mention the findings whose scans found nothing (e.g., "Per-finding cascade scans: findings 2 and 5 produced no cascades."). The line is procedural, not a status row — it confirms the scan was run, parallel to the cross-skill scan's "Scanned via grep — no inconsistencies" announcement. **Mixed-outcome batches**: when a batch contains findings with different cascade-scan outcomes — some producing no hits, others producing complementary hits (per the Complementary-hit confirmation pattern sub-rule below), others producing primary-site confirmations, and/or others producing actual cascades — use the per-finding form (`Finding N: <count> hits found in <file-list>, confirmed complementary (<reason>); no cascade.` for complementary hits; `Finding N: <count> hits found, all accounted for as primary sites ...; no extraneous cascade.` for primary-site confirmations; `N.cascade` rows for actual cascades) for EACH finding whose scan produced hits, AND the per-batch `Per-finding cascade scans: findings X and Y produced no cascades.` line covering ONLY the no-hits findings. The per-batch line is reserved for the no-hits case; do NOT collapse complementary-hit or primary-site-confirmation findings into it — each non-empty outcome takes its own per-finding line. This preserves audit-trail reconstruction: a reader must be able to enumerate each finding's scan outcome from the summary, and compressing complementary hits into "no cascades needed" hides the fact that hits were found and reviewed.

  **Primary-site confirmation pattern**: A scan may find hits that are ALREADY-PLANNED primary edit sites rather than new cascades. This is distinct from both "scan found nothing" and "scan found cascade N.cascade." Example: a terminology-harmonization finding (Improvement N) is planned as N parallel instantiations (`N.a`, `N.b`, ...); the cascade scan for that finding greps the same term and finds exactly those N sites — no extras. Another example: a primary edit's cascade scan for related terminology finds one hit, but that hit IS the co-equal primary already keyed elsewhere in the summary. Report this pattern as `Finding N: <count> hits found, all accounted for as primary sites (<optional brief site list>); no extraneous cascade.` This preserves audit-trail visibility that the scan ran AND that the hits were reviewed and confirmed as planned primaries, without creating spurious `N.cascade` rows or misreporting "scan found nothing" when it actually found the expected sites.

  **Complementary-hit confirmation pattern**: A scan may also find hits that are NEITHER planned primary sites NOR cascade requirements — they are related existing content that happens to be consistent with (not modified by) the planned change. This is distinct from all three prior outcomes: "scan found nothing" (no hits), "primary-site confirmation" (hits are planned primaries), and "scan found cascade N.cascade" (hits require mirroring the primary). Example: a finding that tightens a rule in one section scans related terminology across the file and finds the same terminology used correctly in a sibling section where the existing text remains valid under the new rule — the sibling text is complementary, not a primary site, not a cascade target. Report this pattern as `Finding N: <count> hits found in <file-list>, confirmed complementary (<one-phrase reason, e.g., "definitional comment, not operational rule" or "parallel register, already consistent with the new rule">); no cascade.` This preserves audit-trail visibility that the scan ran and that the hits were reviewed and verified consistent, distinguishing from (a) the "no hits" per-batch line (scan found nothing at all), (b) primary-site confirmation (hits ARE the planned primaries), (c) cascade rows (hits require modification), and (d) restructure-scope pattern (Feature IS the scan scope; see sub-rule below). The six scan-outcome patterns together — no-hits / primary-site / complementary-hit / cascade / restructure-scope / scope-narrowed-cascade — cover the full space of scan results and keep the audit trail reconstructible.

  **Restructure-scope pattern**: For Feature-level architectural restructures whose scope IS the target's surface reorganization (new file creation, extraction to `references/`, file split, consolidation), the per-finding cascade scan concept is circular — the entire target's surface is affected by the restructure itself, so "scan for related text affected by the primary edit" does not meaningfully narrow. Report as `Finding N: restructure scope self-contained; encompassed-findings' primary-edit scans are reported separately under their own finding rows.` This is distinct from no-hits (no scan was warranted because nothing was textually local) and from cascade (where a primary edit needs mirror fixups elsewhere). The restructure IS the scan scope; do NOT collapse the Feature into the per-batch no-hits line, which is reserved for findings whose primary edit is textual-local. Each encompassed finding still carries its own per-finding scan line under its own finding number, per the Restructure-encompassing findings sub-rule in Edit ordering.

  **Quick-reference — cascade-scan outcome by shape** (parallel to the Quick-reference row-keying table for post-implementation summary rows below; this table covers the six outcome patterns described above):

  | Outcome | Condition | Canonical report form |
  |---|---|---|
  | `no-hits` | scan found nothing at all for one or more findings in the batch | per-batch line before the status table: `Per-finding cascade scans: findings X and Y produced no cascades.` (or `...no cascades needed for any of N findings.` when all findings produced no hits) |
  | `primary-site` | hits found ARE already-planned primary edit sites (e.g., `N.a`/`N.b` parallel instantiations of the same semantic change) | per-finding line: `Finding N: <count> hits found, all accounted for as primary sites (<optional brief site list>); no extraneous cascade.` |
  | `complementary-hit` | hits found are related existing content that is consistent with (not modified by) the planned change | per-finding line: `Finding N: <count> hits found in <file-list>, confirmed complementary (<one-phrase reason, e.g., "definitional comment, not operational rule" or "parallel register, already consistent">); no cascade.` |
  | `cascade` | hits found require mirror edits to preserve consistency with the primary change | `N.cascade` row in post-implementation summary status table, with reason naming touched location + substantive change |
  | `restructure-scope` | Feature-level architectural restructure IS the scan scope (encompassed findings' own scans are reported under their own finding rows) | per-finding line: `Finding N: restructure scope self-contained; encompassed-findings' primary-edit scans are reported separately under their own finding rows.` |
  | `scope-narrowed-cascade` | scan found sibling cascade-target candidates initially; per-sibling investigation revealed categorical-different surface shape / absent-parallel section / sibling-coherence-breaking content; scope narrowed per Mid-implementation cascade scope-narrowing sub-rule (Guardrails §Cross-skill consistency) | per-finding line: `Finding N: <count> cascade-target candidate(s) found in <sibling-list>; per-sibling investigation narrowed scope per Mid-implementation cascade scope-narrowing rule (<one-phrase reason per excluded candidate>); no cascade applied.` Distinct from `no-hits` (scan found nothing at all) and from `cascade` (hits required mirror edits); the scan DID find hits and the scope-narrowing IS the outcome. Pair with the Cross-Skill Scan section's fourth form (audit phase) or the post-implementation summary's `scope narrowed per Mid-implementation cascade scope-narrowing rule` line (implementation phase) to keep the audit trail complete. |

  Rule of thumb: every finding produces exactly one outcome from this table. When reporting a mixed-outcome batch (per the Mixed-outcome batches sub-rule above), compose per-finding lines for each non-`no-hits` outcome, and a single per-batch line for the `no-hits` findings. Do NOT collapse `primary-site` or `complementary-hit` outcomes into the per-batch `no-hits` line — each requires its own per-finding line so the audit trail records that hits were found and reviewed.

  **Multi-outcome per finding**: when a single finding's scan produces hits that split across multiple outcome patterns (common case: a terminology-harmonization finding with parallel primary instantiations whose scan matches both the planned sites AND related non-target phrasing that stays consistent), compose a combined per-finding line that names each outcome with its hit count and its brief reason: `Finding N: <total> hits — <count-A> <outcome-A> (<reason-A>); <count-B> <outcome-B> (<reason-B>); <verdict — "no cascade" or "N cascade rows follow">.` The rule-of-thumb "exactly one outcome per finding" above applies when the scan's hits fall cleanly into one pattern; multi-outcome scans preserve the per-outcome breakdown so the audit trail records that hits of each kind were found and reviewed. Distinct from the Mixed-outcome batches rule above (which governs different findings in the same batch each having their own single outcome); this sub-rule governs one finding whose own hits span multiple outcomes. Multi-outcome per finding is a **composition** over the six canonical outcome patterns, not a seventh pattern — the enumeration in the Complementary-hit confirmation pattern sub-rule remains complete. Worked instantiation: a terminology-harmonization finding applied at 4 parallel sites (`N.a`/`N.b`/`N.c`/`N.d`) scans for the same phrase across the file and finds 8 hits — 4 at the planned primary sites (`primary-site` outcome) and 4 at related phrasing that uses the same vocabulary but references different semantics (`complementary-hit` outcome); reports as `Finding N: 8 hits — 4 primary sites (lines L1/L2/L3/L4 — the planned N.a/N.b/N.c/N.d instantiations); 4 complementary (lines M1/M2/M3/M4 — related phrasing with different semantics, unchanged under the finding's rule); no extraneous cascade.` A 3-outcome instantiation demonstrates that the composition scales to any arity: a finding whose scan produces 1 planned primary site + 1 cascade-target hit (keyed N.cascade) + 1 complementary hit on unrelated-but-consistent existing content reports as `Finding N: 3 hits — 1 primary site (L1 — the planned Step 5 prose edit); 1 cascade (L2 — Process Flow diagram, keyed N.cascade); 1 complementary (L3 — step-name reference in HARD-GATE bullet; no per-message constraint in prose); 1 cascade row follows.` The 3-outcome form extends the 2-outcome pattern by appending a third `<count> <outcome> (<reason>)` clause before the verdict — no structural change to the canonical form.

  **Cascade-scan discipline — search for semantic variants, not just exact strings**: When the primary change is a count, a renamed term, or a structured label, grep BOTH the exact old string AND its semantic variants. For a count change (e.g., "11 → 12"), also search for the number rendered elsewhere: tabular summary rows, alignment tables, process-flow diagrams, prose paraphrases that state the count in different phrasing, ordinal forms ("twelfth"), and word-form spellings ("twelve"). For a renamed term, also search for its plural, its adjective form, hyphenated variants, and compound constructions. For a structured label (e.g., `cf_id`), also search for prior names it may have used, its singular/plural siblings (`cf_ids`, `cf_refs`), and prose references that name the concept without the exact label. For **new cross-references** — text that introduces a reference to a section name, field name, or file path that the original edit did not itself define (e.g., adding "Record this in the audit report's Parameters Interpretation Note" where that named section may not exist in the target's template) — there is no "old string" to grep for, so the scan shape inverts: for each new reference the edit introduces, verify during per-finding planning that the referent RESOLVES AT EXACT-STRING-MATCH against existing content in the target skill's shipped files (SKILL.md + templates/ + references/) — including capitalization, punctuation, whitespace, and header-hash count for markdown anchors; key name and quote style for YAML keys. Near-miss resolution — conceptually valid but lexically divergent (e.g., `§Place-type / polity composite handling` vs the actual `§Place-type / Polity Composite Handling` heading) — fails grep discoverability in future audits and forces a verification-backstop rework cycle. When the new cross-reference targets content YOU are creating in the same batch, draft the referent text FIRST (including its exact capitalization and punctuation), then copy-paste the exact string into the reference to eliminate drift. If it doesn't resolve, choose one of three repairs at planning time: (a) rename the new reference to an existing referent, (b) add the referent (as a template section or file, with its own cascade row), or (c) soften the reference to name acceptable alternative placement ("either as a standalone Section X or folded into Section Y"). Catching this at planning time prevents the verification backstop from being the primary detection mechanism for this shape. For **structured-template targets** (proposal cards, batch manifests, dossiers — any file combining YAML frontmatter with a markdown body that renders the frontmatter concepts in tables, formulas, or worked examples), scan BOTH registers on any frontmatter schema change: (i) the frontmatter enum / field definition that the primary edit targets; AND (ii) the body-level rendering sections where the field is exemplified in tables, formulas, prose paraphrases, or worked examples. A frontmatter enum expansion without body-section update leaves the template internally inconsistent — the schema and its rendering disagree — and the body rendering IS part of the template's schema surface, not decorative explanation. Common two-register miss: adding a new value to a YAML enum in frontmatter, then discovering at verification that the template body has a table row and/or a formula string referencing the prior enumeration; both need updating to match the new schema. The verification backstop catches these misses, but per-finding scanning should aim to catch them at planning time. Over-scan is cheap; under-scan pushes the miss to the verification backstop, where the skill explicitly disallows it as a substitute for per-finding scanning.

  **Post-implementation summary**: After all edits, present a summary table or list showing the status of each finding: "implemented", "superseded by finding N", "cascade from finding N", "co-edit with finding N", or "skipped (reason)". Additionally, for meta-findings whose suggestion is realized by other findings' primary edits rather than by a dedicated edit, use `implemented (realized via findings M, N, ...)` naming the contributing findings explicitly. This gives the user a clear per-finding status rather than requiring them to infer outcomes from the edit sequence.

  **Severity-count line (optional)**: after the status table, you MAY include a one-line count summary of the form `N audit findings → N primary edits + N cascades = N edit rows; M CRITICAL / X HIGH / Y MEDIUM / Z LOW combined`. Cascade rows inherit severity from their originating finding for counting purposes. Sub-keyed primary rows (`N.a`, `N.b`, `N.c`) each inherit severity from the parent finding for counting purposes, parallel to cascade rows — the M/X/Y/Z distribution reflects edit-row severity, not finding severity, so a finding with 2 co-equal primaries and 1 cascade contributes 3 rows to the distribution (not 1). Discarded rows (`now-moot — discarded`) appear in the status table for audit-trail visibility but are excluded from the severity-count line's edit-row distribution — count only rows that represent actual edit work (primary + sub-keyed primary + cascade), not rows that merely record that a finding was retracted during re-evaluation. The line is optional — omit when the status table's row count is small enough to enumerate at a glance (≤4 rows); include when the table has 6+ rows or when mixed-outcome batches produce cascade+primary combinations worth aggregating. The canonical decomposition (audit findings → primary edits + cascades = edit rows) preserves audit-trail reproducibility better than a single edit-row count, and is the recommended default; alternative shapes are acceptable when internally consistent but should preserve the audit-findings vs primary-edits vs cascades decomposition (e.g., `13 audit rows → 13 edit rows = 13 edit rows` collapses three distinct counts — 6 findings, 10 primaries, 3 cascades — into a single conflated count and loses the decomposition value the canonical form preserves). Shape is a judgment call; consistency within a single report matters more than a specific count schema.

  **Quick-reference — row-keying by shape** (consult the Worked example block below for edge cases and rationale):

  | Shape | Row keys | Status field |
  |---|---|---|
  | 1 finding → 1 edit | `N` | `implemented` |
  | 1 finding → multiple co-equal parallel edits (same semantic change applied at N sites) | `N.a`, `N.b`, ... | `implemented` on each (do NOT add co-edit notation) |
  | 1 finding → primary edit + cascade(s) | `N` + `N.cascade` (or `N.cascade-a` / `N.cascade-b` for multiple cascades) | `implemented` on primary; `cascade from finding N — <reason>` on cascade rows |
  | Multiple findings → one shared edit | one row per finding, each keyed by its own number | `implemented (co-edit with finding M)` on each row, naming partners |
  | Meta-finding → no dedicated edit; realized by other findings' primary edits | `N` (finding keyed by its own number; no sub-keys) | `implemented (realized via findings M1, M2, ...)` naming the contributing findings |
  | Feature-level architectural restructure encompassing other findings' content | `N` | `implemented — <restructure description>; encompasses findings M1, M2, ...` naming the findings whose content physically lives in the new structure |

  Rule of thumb: condensing multiple parallel edits into a single summary row with a bulleted list of sites is non-compliant — use one row per edit. The sub-letter keying IS the co-equality signal; do not add redundant co-edit notation to it.

  List cascade edits as separate table rows keyed `N.cascade` (where N is the originating finding's number); the status column contains `cascade from finding N — <concise reason: one or two lines max, naming both the touched location and the substantive change>`. If a single finding requires multiple primary edits in different sections of the target skill (neither edit is a cascade of the other — both are co-equal owners of the suggestion), key them `N.a`, `N.b`, etc. Reserve `N.cascade` for edits that merely keep related text consistent with a primary change. When two or more separate findings are resolved by a single shared text edit (their Suggestions collapse onto the same passage), keep each finding as a distinct row in the summary table — key each by its original finding number and use the status `implemented (co-edit with finding M)` on each row to name the partners. Do NOT re-key them as sub-letters; `N.a`/`N.b` is reserved for one-finding-multiple-edits only. This preserves the finding-to-edit relationship so a reader of the status table can reconcile it against the audit's finding count. This keeps per-edit granularity consistent across audits.

  Worked example — status field by case:
  - Baseline (N findings, N independent edits): row `Improvement 1` → status `implemented`; row `Improvement 2` → `implemented`; row `Improvement 3` → `implemented`. No sub-lettering, no co-edit notation, no cascades. This is the default shape when each finding has its own independent text change and nothing cross-cuts.
  - One finding, three edits: row `Improvement 1.a` → status `implemented`; row `Improvement 1.b` → `implemented`; row `Improvement 1.c` → `implemented`. The keying itself signals co-equality; do NOT add `(co-edit with Improvement 1.b/1.c)` to the status field.
  - One finding, N parallel instantiations of the same semantic change (no primary): row `Improvement N.a` → `implemented`; row `Improvement N.b` → `implemented`; ...; row `Improvement N.g` → `implemented`. When a single finding's suggestion ("standardize phrasing X across the file") is applied to multiple parallel locations of the same target text, the edits are co-equal — none is "primary" with the others as "cascades." `N.cascade` requires a primary semantic change at one location with related-text fixups elsewhere; parallel instantiations of the same change lack that primary/cascade relationship and use `N.a`/`N.b`/... instead. Rule of thumb: if you can swap the order of any two edits without changing the meaning of the others, they are parallel co-equals, not primary-plus-cascade.
  - Three findings, one shared edit: row `Improvement 1` → `implemented (co-edit with Improvements 4, 7)`; row `Improvement 4` → `implemented (co-edit with Improvements 1, 7)`; row `Improvement 7` → `implemented (co-edit with Improvements 1, 4)`. Each row names the other partners.
  - One finding, two cascades in different sections: row `Improvement 3` → `implemented`; row `Improvement 3.cascade-a` → `cascade from Improvement 3 — <reason A>`; row `Improvement 3.cascade-b` → `cascade from Improvement 3 — <reason B>`. The `cascade-a / cascade-b` suffixes mirror the `N.a / N.b` pattern for primary edits and preserve per-edit granularity. When two cascades from the same finding are tightly coupled (the same conceptual change in adjacent text), it is acceptable to merge into a single `N.cascade` row instead.
  - One finding, two co-equal primaries with one cascade each: row `Improvement N.a` → `implemented`; row `Improvement N.b` → `implemented`; row `Improvement N.cascade-a` → `cascade from Improvement N — <reason dependent on N.a>`; row `Improvement N.cascade-b` → `cascade from Improvement N — <reason dependent on N.b>`. When a finding has multiple co-equal primaries AND each primary has its own cascade(s), compose the `N.a/N.b` pattern with the `N.cascade-a/N.cascade-b` pattern. The `cascade-a`/`cascade-b` suffixes DO NOT have to align with the primary suffixes — a cascade's letter ordering reflects its own dependency position, not the primary it follows. Name the dependency explicitly in the reason field (e.g., "cascade from Improvement N — mirrored to sibling template per structural-identity claim; follows N.b primary") so a reader can reconstruct which primary each cascade depends on. Worked instantiation: a finding whose suggestion applies to two co-equal locations (the target skill's main file AND a paired sibling template) where each primary has its own dependent fixup — the main-file primary (`N.a`) has a cascade in a validation section (`N.cascade-a`) that cross-references the new content, and the template primary (`N.b`) has a cascade in a structurally-paired sibling template (`N.cascade-b`) that mirrors the change per a structural-identity claim. Either cascade could be applied before or after its companion primary; what matters is that each cascade's reason field names the primary it depends on so the reader can reconstruct the dependency graph from the status table alone.
  - One finding, two co-equal primaries with a single cascade from one primary (asymmetric): row `Improvement N.a` → `implemented`; row `Improvement N.b` → `implemented`; row `Improvement N.cascade` → `cascade from Improvement N — follows N.a primary; <substantive reason naming touched location and change>`. When a multi-primary finding has exactly one dependent cascade rooted in only ONE of the primaries (e.g., verification-backstop catch that softens a new cross-reference introduced by `N.a` without touching `N.b`'s section), the single-suffix `N.cascade` key is acceptable PROVIDED the reason field explicitly names which primary the cascade follows ("follows N.a primary" / "follows N.b primary"). The letter-suffix `N.cascade-a` form is reserved for the multi-cascade case above; using it for a single cascade would imply an absent `N.cascade-b` and mislead the reader. Worked instantiation: a finding whose two primaries split across two phases (`N.a` in Phase 0, `N.b` in Phase 2) where only the Phase 0 primary introduced a new cross-reference that required a follow-up softening — the softening is one cascade, rooted in `N.a`, keyed `N.cascade` with reason "follows N.a primary; Phase 0 reference softened to name alternative placement." Decision rule: number of cascades determines suffix. Zero cascades → no cascade row. Exactly one cascade → `N.cascade` with reason naming the dependency. Two or more cascades → `N.cascade-a` / `N.cascade-b` / ... per the paired-cascade example above.
  - Two findings, same file but different lines: row `Improvement 2` → `implemented`; row `Improvement 5` → `implemented`. Both rows are baseline `implemented`; do NOT use co-edit notation. Co-edit is reserved for a single shared text edit resolving multiple findings, not separate edits at different locations in the same file. If a reader wants to see that two findings touched the same file, that relationship is discoverable from the status-row reasons (which name the touched locations), not from the status field itself. Attempting to repurpose co-edit for same-file-different-lines conflates "shared text" with "shared file" and breaks the reader's ability to reconcile row counts against the audit's finding count.
  - Two findings, adjacent passages batched into one Edit tool call: row `Improvement 2` → `implemented`; row `Improvement 3` → `implemented`. Both rows are baseline `implemented`; do NOT use co-edit notation. A single Edit tool call can replace one old_string with a new_string that inserts two or more distinct semantic blocks (e.g., a definition list for finding A and a table for finding B at the same insertion point). Tool-call granularity is NOT the co-edit criterion — passage granularity is. If the two new blocks could be described in independent sentences ("Improvement 2 added the family enumeration; Improvement 3 added the user-label mapping table"), they are separate edits despite being co-located and tool-batched. Co-edit is reserved for cases where one continuous passage simultaneously serves multiple findings (e.g., a subsection whose prose covers two findings in one breath, or a table whose rows collectively answer two findings). Rule of thumb: if you can point at distinct text fragments and say "this fragment resolves A, that fragment resolves B," they are separate edits; if the same fragment resolves both, it is co-edit.
  - Two cascades from different findings batched into one Edit tool call: row `Issue N.cascade` → `cascade from Issue N — <reason>; batched in same Edit with Issue M.cascade`; row `Issue M.cascade` → `cascade from Issue M — <reason>; batched in same Edit with Issue N.cascade`. Both cascade rows stay keyed under their own originating findings; the batching is captured in the reason field, parallel to the primaries-batched pattern in the preceding example. Do NOT re-key as a single shared `N+M.cascade` row — that breaks the finding-to-edit reconcilability the keying preserves. Decision rule: batching is a tool-call optimization, not a keying criterion. Whether two cascades are in one Edit or two, each cascade row stays under its originating finding's number; the only thing that changes is whether the reason field names the batch partner. Worked instantiation: during canon-addition audit implementation, the SKILL.md Output section edit served BOTH Issue 1.cascade-a (clarificatory-retcon skips new-CF artifact — cascading from Issue 1's Clarificatory-Retcon Variant primary in accept-path.md) AND Issue 3.cascade (attribution-form extension — cascading from Issue 3's Placement convention primary in accept-path.md); keyed as two separate cascade rows with `batched in same Edit with Issue X.cascade` notes in each reason field. Distinguishes from co-edit (one passage resolves multiple findings — not applicable here; the Output section edit has distinct fragments for each cascade's concern) and from meta-finding (no dedicated edit — not applicable here; each cascade has its own identifiable fragment).
  - Cross-skill cascade vs cross-skill parallel: when a primary change in the target skill MUST be mirrored to a sibling skill's parallel file to preserve a stated structural-identity claim (e.g., "structurally identical to `<sibling>/templates/...`"), the sibling's edit is a cascade — key `N.cascade` (single mirror) or `N.cascade-a` / `N.cascade-b` (multiple mirrors). The sibling edit lacks primary semantic authority; it follows the target. In contrast, when a Finding's Suggestion explicitly names the sibling as a co-equal target ("update both paired templates"), the sibling edits are parallel primary edits — key `N.a`, `N.b`, `N.c`, `N.d` across all co-equal locations. Decision rule: if the sibling edit's content is dictated by the target edit's content (the sibling just copies the new text), it is a cascade; if both edits instantiate a shared semantic change independently of each other, they are parallel primary edits. When in doubt, prefer cascade keying — it preserves the finding-to-primary-edit relationship and makes the target-skill's authoritative role explicit.
  - New reference file creation with required intra-skill pointers: row `Feature N` → `implemented`; row `Feature N.cascade-a` → `cascade from Feature N — pointer in SKILL.md <section>: added "see <new-file-name>" cross-reference`; row `Feature N.cascade-b` → `cascade from Feature N — pointer in references/<sibling-reference>.md <section>: added "see <new-file-name>" cross-reference`. When a finding creates a new reference file (or template, or any content file that other files must discover via cross-reference for the new content to be load-bearing), the file creation is the primary edit and each cross-reference pointer added in existing files is a **cascade**, keyed `N.cascade` (single pointer) or `N.cascade-a` / `N.cascade-b` / ... (multiple pointers). The pointer edits lack primary semantic authority — their content (a ~30-word sentence naming the new file and its purpose) is dictated by the new file's existence, name, and purpose, which the primary edit established. Decision rule instantiation: if you removed the new-file-creation primary, the pointer edits become meaningless (there is nothing to point at); if you removed one pointer but kept the primary, the other pointer and the primary still make sense — the pointers depend on the primary, not on each other. This is an **intra-skill** cascade shape (same skill's SKILL.md + references mirrored by the new file's existence), distinct from the cross-skill structural-identity-claim case in the preceding example. Worked precedent: canon-addition PA-0017 Feature 1 — new file `references/phase-15a-checkpoint-grep-reference.md` (primary) + cross-reference pointer in SKILL.md §Phase 15a write-phase progress discipline (`Feature 1.cascade-a`) + cross-reference pointer in `references/accept-path.md` §Phase 15a sub-step 4 (`Feature 1.cascade-b`). Decision rule: parallel primaries (`N.a` / `N.b`) would apply only if the Finding's Suggestion named two or more file creations as co-equal outputs; here the Suggestion named ONE new file plus pointers-as-byproduct, so primary-plus-cascades is correct.
  - Meta-finding resolved by other findings' primary edits: row `Issue 1` → `implemented (realized via Improvement 1 and Issue 2)`; row `Improvement 1` → `implemented` (baseline); row `Issue 2` → `implemented` (baseline). When a finding's suggestion is strategic or architectural (e.g., "inline load-bearing content," "unify terminology across sections," "reduce duplication") and its implementation is emergent from the primary edits of OTHER findings rather than a dedicated edit of its own, use this shape. The row uses the finding's own number with no sub-keys — it has no dedicated edit to key. Distinct from co-edit: co-edit requires one shared passage resolving multiple findings; here multiple DIFFERENT passages, each for its own finding, collectively resolve the meta-finding. Decision rule: if finding N has no text uniquely attributable to it, and the edits that resolve N are all primary edits for other findings, use the meta-finding shape. If finding N has at least one text fragment uniquely attributable to it (even if tiny), use `N.a` + co-edit or `N.cascade` instead. The status's parenthetical MUST name every contributing finding by its section-prefixed key so the audit trail is reconstructible from the status table alone.
  - Feature-level architectural restructure encompassing other findings' content: row `Feature 1` → `implemented — 7 references files created + SKILL.md rewritten as thin entry point; encompasses Issues 1-3 and Improvements 1-5 whose content lives in the new references/` (baseline with descriptive reason naming the encompassed findings); row `Issue 1` → `implemented — primary edit in references/preflight-and-prerequisites.md §Oversize-File Protocol` (baseline with location pointing to new file); row `Improvement 3` → `implemented — primary edit in references/phases-7-8-validate-and-commit.md §HARD-GATE Response Mapping` (baseline with location pointing to new file). When a Feature's suggestion IS an architectural restructure (file extraction, new directory creation, file split or merge) and that restructure physically houses other findings' content, key the Feature with its own number and use the status `implemented — <restructure description>; encompasses findings M1, M2, ...` naming the encompassed findings explicitly; each encompassed finding stays keyed by its own section-prefixed number with its status pointing to the specific new file path its content lives in. Decision rule: distinct from meta-finding shape — meta-findings have NO dedicated edit (status says `realized via findings ...`), while restructure-encompassing Features HAVE a dedicated architectural edit (the file creation / SKILL.md rewrite itself) plus encompassed content. Distinct from co-edit — co-edit requires a single shared passage resolving multiple findings; here the Feature's architectural edit (creating the new files) and each encompassed finding's content edit (the actual text inside each file) are physically distinct fragments living in different new files. The Feature's status-row reason explicitly names the encompassed findings so a reader of the status table can reconstruct the encompassing relationship without reading the edited files.

  **Post-edit verification**: After all edits are applied, re-read each edited file — full file for short files; targeted Reads of edited regions with flanking context for longer files — and verify as a single pass. For numbering-continuity check 2(a) on files >150 lines, prefer grep per that check's guidance; targeted-read coverage of each edited region satisfies the remaining checks. A strict end-to-end re-read of an unchanged 400-line file is not required when the verification-check items can be satisfied from the edited regions alone.

  **"Each edited file" includes cascade targets.** Intra-skill cascade targets (`references/*.md`, `templates/*.md`, and similar files under the target skill's own directory) and cross-skill cascade targets (sibling-skill files edited per `N.cascade` rows in the summary) BOTH count as edited files for verification purposes. Apply checks 1-5 below to every cascade target just as you would to the target skill's SKILL.md; check 6 remains meta-tooling-gated for the target only. A malformed cascade edit to a sibling reference file is otherwise silent — it will not surface at verification if verification is interpreted as "target-only" — and the failure mode (broken cross-reference in a sibling skill that will only trigger on that sibling's next invocation, possibly weeks later) is exactly the kind of drift the cascade discipline exists to prevent.
  1. **No overlap or contradiction** — edits don't conflict with each other
  2. **Cross-references valid**:
     - (a) **Numbering continuity** — step, phase, and section numbers are sequential with no gaps or duplicates. If the file has >150 lines OR >10 numbered references across multiple levels AND the file uses a single consistent numbering convention, prefer grep pattern search (e.g., grep for `Step [0-9]`, `### [0-9]`) to confirm. If the file mixes numbering conventions (numbered top-level steps combined with checkbox items, numbered sub-lists, and bulleted items in the same file — no single grep pattern catches all of them), visual scan via targeted Reads of each numbered block is acceptable. For shorter files with consistent conventions, a visual scan suffices. Adapt grep patterns to the target skill's convention (numbered items, lettered sub-steps, or markdown headers).
     - (b) **File paths valid** — all referenced file paths still exist and point to correct targets.
     - (c) **New cross-references** — references introduced by new text point to content that actually exists. When the target skill uses nested numbering (sub-steps within steps), verify that cross-references disambiguate between levels (e.g., "Step 1, sub-step 5" vs. "Step 5").
     - (d) **Overview diagrams** — high-level overviews that become slightly inaccurate due to new branching logic are acceptable if the detailed step text handles the nuance. Note the discrepancy but do not force-update overview text that would become harder to scan.
  3. **Sequential flow coherent** — the skill reads coherently end-to-end after all edits
  4. **Contextual consistency** — numbering, terminology, and cross-references are consistent with adjacent unchanged text
  5. **Frontmatter integrity** — if any edit touched the YAML frontmatter, verify `---` delimiters are intact and the YAML parses correctly (name, description, and arguments are present and properly quoted)
  6. **Cross-skill sibling scan** (meta-tooling targets only) — if the target is a meta-tooling skill (brainstorm, skill-creator, skill-audit, or similar process/tooling skill) AND any edit touched shared terminology, conventions, or tag semantics (per Guardrails §Concrete shared-surface triggers), confirm the sibling scan has been run and documented in the post-implementation summary. Skip this check when the target is not a meta-tooling skill, OR when no edits touched shared surfaces. This is the gate that catches the "Extends to user-directed edits on meta-tooling skills" rule before it slips — the scan itself lives in Guardrails §Cross-skill consistency, but the confirmation lives here.

  If any check fails, fix the offending edit(s), then re-run the full verification pass. Do not selectively re-check — a fix in one area can introduce issues in another.
- **Cross-skill consistency** — If the target skill is part of a multi-skill workflow AND any finding affects interfaces shared with sibling skills, scan sibling skills for inconsistencies. Report cross-skill inconsistencies as Issues. Skip when all findings are internal to the target skill.

  **Two distinct sibling-skill operations are governed by this guardrail — they are independent and either, both, or neither may apply on any given implementation:**

  - **Cross-skill cascade** applies to ANY target skill (meta-tooling or canon-mutating) when a stated structural-identity claim in the target (e.g., a template comment reading "structurally identical to `<sibling>/templates/...`") makes a sibling edit mandatory to preserve the claim. Keyed `N.cascade` in the post-implementation summary; sub-rule lives in the Quick-Reference row-keying section under `Cross-skill cascade vs cross-skill parallel`. The sibling edit follows the target edit's content — no independent drift-check is needed because the content is dictated.
  - **Cross-skill sibling scan** applies ONLY to meta-tooling targets (brainstorm, skill-creator, skill-audit, skill-consolidate, and similar) as a defensive drift-check across sibling meta-tooling skills. Runs grep / read / diff across meta-tooling siblings to confirm the change doesn't leave sibling skills inconsistent with the target. Scan method must be named in the announcement; an unexecuted scan must be announced as `Scan not performed — <reason>`, not fabricated as executed.

  A single implementation may require cross-skill cascade, cross-skill sibling scan, both, or neither. The two are operationally independent: cascade is about mirroring dictated content to preserve a claim; scan is about verifying the absence of drift. Confusing them — escalating a simple cascade into a full sibling-scan protocol, or skipping a mandatory cascade because "scan is meta-tooling only" — has been a real source of friction. The sub-rules below elaborate each operation.

  **Concrete shared-surface triggers**: A finding affects a shared surface when it changes any of the following:
  - `docs/plans/*.md` design-doc output path or frontmatter conventions (produced by `brainstorm`)
  - `docs/triage/*.md` triage-file format (produced by `brainstorm` for multi-deliverable triages)
  - `brainstorming/*.md` canon-pipeline proposal structure (produced by `skill-creator` in fresh mode; consumed by `skill-creator` in compile mode)
  - `archive/*/` archival destination paths or naming conventions
  - `specs/*.md` or `tickets/*.md` structure and numbering conventions — worldloom does not yet ship with these directories; check for presence before flagging, and if absent, skip this bullet
  - Shared terminology that appears in multiple skills
  - Output format consumed by a downstream skill in the same pipeline (e.g., `brainstorming/*.md` proposals consumed by `skill-creator`; `docs/plans/*.md` design docs consumed by follow-up implementation skills)
  - Shared template files referenced by multiple sibling skills — especially when a template carries an explicit "structurally identical to `<other-skill>/templates/...`" claim or comment. Schema drift between such templates is the failure mode: if finding N adds, removes, or repurposes a field in one skill's template, the sibling's matching template needs the same change to preserve the structural-identity claim. Catch this at per-finding planning time, not via the post-edit verification backstop.
  - **Consumer-side output-schema dependencies** — when a finding introduces references to specific field names, section names, or structural conventions produced by a sibling skill's output (e.g., a character-dossier section name like "Epistemic Position §Known firsthand", a canon-addition CF-id format, a proposal-card frontmatter field) — the reference is a schema dependency on the sibling, even when the target skill is NOT meta-tooling and the finding makes no explicit structural-identity claim. This sits between Cross-skill cascade (which requires a stated identity claim in the target) and Cross-skill sibling scan (which is meta-tooling-gated). For non-meta-tooling targets, verify the referenced schema against the sibling's current template with a point-of-reference read (scan method (b) Read, or (d) Diff when the reference is structured) rather than a full sibling scan. **For multi-term existence verification** — confirming a sibling template exposes several named fields or a named taxonomy in a single check — method (a) Grep is an acceptable alternative to Read or Diff: grep for all referenced terms in one call, confirm the sibling exposes them, skip the full-file read. Prefer Read or Diff when the verification is about field-level semantics (what a field means, how it's populated, what values it takes) rather than field-level existence (whether the field is exposed at all). Announce the verification with the same hypothesis-first pattern: "Investigating whether `<sibling-skill>`'s current template still exposes `<field-or-section-name>`: reading `<sibling-path>` to verify." (or `grepping for <terms>` when using method (a) for multi-term existence). Record the verification outcome in the Cross-Skill Scan section of the audit report using the existing first-form ("Scanned: `<sibling-path>` via read — schema confirmed, no finding adjustment needed") or second-form ("Scanned: `<sibling-path>` via read — sibling's section has been renamed to X, adjusted finding wording") format — substitute "via grep" in place of "via read" when the method was grep.
  - **Documented alignment-convention statements** — when one skill's documentation explicitly declares a convention that siblings "should be kept aligned on", "mirror", or "follow the same pattern as", treat any finding that touches the conventioned content as a shared-surface trigger. The alignment statement itself is the interface; the conventioned content is the shared surface. Example: `canon-addition/SKILL.md` §Pre-flight Check Large-file method states *"The three skills should be kept aligned on the convention; future canon-addition runs that discover additional anchor patterns for new file shapes should update this section so siblings can mirror"* — any finding adding to the anchor-pattern list triggers a cross-skill cascade planning step at audit phase, even when the target skill is non-meta-tooling. Distinct from the "Shared template files with structural-identity claim" trigger above (which covers template-level content mirroring via an explicit "structurally identical to `<other-skill>/templates/...`" comment); alignment-convention statements cover SKILL.md / reference-file content conventions that are declared-shared-by-prose rather than declared-identical-by-template-comment. Audit-phase handling: include the Cross-Skill Scan section with the third form (`Cascade planned for implementation: <sibling-file> per <alignment-convention statement quote>`) OR, if the auditor investigates siblings during the Auxiliary investigation window and scope-narrows, the fourth form (`Scanned: <list> via <method> — cascade-target candidates found; per-sibling investigation narrowed scope (<reason>); no sibling cascade planned.`). Catching this at audit phase prevents the implementation-phase mid-flow discovery pattern where the shared-surface impact first surfaces during per-finding cascade scanning.

  When triggered, list the scanned sibling skills in the audit report alongside what (if anything) was adjusted. If no inconsistency was found, state "Scanned: <skill list> — no inconsistencies." This makes the scan auditable. When a finding identifies a required cross-skill cascade but no sibling SCAN was run during the audit phase (typical of non-meta-tooling targets where the sibling-scan operation doesn't apply), use the Cross-Skill Scan section's third form — "Cascade planned for implementation: <sibling-file> per <structural-identity claim>" — and keep the cascade itself as an `N.cascade` row in the post-implementation summary once implementation runs.

  **Mid-implementation cascade scope-narrowing**: If per-sibling investigation at implementation time reveals that a planned cascade target has a categorically different surface shape than the audit assumed — for example, the sibling already handles the same concern via a different mechanism (a different section structure, a different reference-file split, an inline method the audit did not notice), or the sibling's paralleled section is absent entirely, or cascading the target's content would break the sibling's own internal coherence — it is acceptable to narrow the cascade scope. Announce the scope-narrowing explicitly in user-facing text before applying the remaining cascades, naming the excluded sibling and the dissimilar-shape reason (e.g., "Cascade scope decision: `canon-addition/SKILL.md` already has a more elaborated Large-ledger method focused on ID-allocation scanning, a different gap-shape than the other two siblings; applying cascade only to the two reference files whose gap exactly mirrors the target's"). Do NOT silently skip divergent cascade targets — the announcement preserves the audit trail that an audit-plan-to-implementation gap existed and was resolved with judgment rather than by oversight, and it lets a future reader reconstruct why the post-implementation summary has fewer `N.cascade` rows than the audit's Cross-Skill Scan section promised. This is distinct from Re-evaluation (which addresses target-file changes between audit and implementation) and from Partial implementation (which addresses user-specified finding subsets); scope-narrowing addresses a planning-time-to-execution-time gap in the CASCADE side of a finding when sibling state differs from the audit's assumption (see §Mid-implementation primary-edit scope-narrowing in §Edit ordering for the parallel primary-side rule that addresses the same gap-shape on the primary side, when target instantiations — not sibling state — differ from the audit's assumption).

  The rule applies symmetrically in two directions. **Audit-over-promised**: the audit identified a cross-skill cascade target (Cross-Skill Scan section's third or fourth form was emitted) and implementation-time per-sibling investigation revealed the planned cascade should be narrowed (categorical-different shape, absent parallel section, sibling-coherence-breaking content). **Defensive-over-discovered**: the audit identified no cross-skill cascade (Cross-Skill Scan section omitted because no shared surfaces were touched), but implementation-time defensive scanning surfaced a candidate that per-sibling investigation then narrowed back to no cascade. Both cases use the same canonical post-implementation line (`Cross-skill sibling scan: Scanned: <sibling-list> via <method> — scope narrowed per Mid-implementation cascade scope-narrowing rule (<reason per excluded sibling>); no sibling cascade applied.`). In the defensive-over-discovered case, the reason field should explicitly note that the candidate was surfaced at implementation time rather than at audit time (e.g., `defensive implementation-phase scan surfaced <sibling> as candidate; per-sibling investigation revealed <reason>`) — this preserves the audit-to-implementation gap as load-bearing evidence rather than burying the over-discovery as if the audit had identified the candidate.

  **Scan methods** — pick one based on the finding's shape:
  - (a) **Grep** for the changed terminology or convention across `.claude/skills/*/SKILL.md` when the finding changes a word-level convention (a tag name, a section heading, a keyword, a status label).
  - (b) **Read** a sibling's SKILL.md end-to-end when the finding touches that sibling's declared interfaces or outputs (inputs the sibling consumes, artifacts it produces, templates it references).
  - (c) **Glob + check** when the finding changes a shared-path convention (`archive/*/`, `brainstorming/*.md`, `worlds/<slug>/`, `docs/plans/*.md`, `docs/triage/*.md`).
  - (d) **Diff** a shared template or structured file byte-by-byte against the sibling's parallel file when verifying a "structurally identical to `<other-skill>/templates/...`" identity claim or detecting schema drift. Prefer this over Read when the two files are parallel enough that line-by-line comparison is more informative than end-to-end reading — common for YAML templates and structured reference files.

  **Scan target — meta-tooling targets only**: When the target is a meta-tooling skill (skill-creator, skill-audit, skill-consolidate, brainstorm, and similar), the scan's audit target is *other meta-tooling siblings*, not *generated skills that instantiate the meta-tooling skill's template* (character-generation, diegetic-artifact-generation, propose-new-canon-facts, etc.). Matches in generated skills are confirmations that the generated skill correctly instantiates the meta-tooling convention — they are NOT sibling inconsistencies. If a grep matches a generated skill, spot-check that the generated usage is consistent with the convention being edited; do not cascade the edit to the generated skill (that's the meta-tooling skill's job at its next invocation, when it re-authors generated files from the updated template).

  Name the method used in the scan announcement so the result is reproducible (e.g., "Scanned `brainstorm`, `skill-creator` via grep for `templates/` references — no inconsistencies" or "Scanned `create-base-world` via read of `templates/*.yaml` + diff against generic source — no inconsistencies"). A scan announcement without a named method is not auditable.

  **The announcement must reflect work actually performed.** Writing "Scanned via grep for X" when no grep was executed violates the audit-trail discipline — even when the auditor's reasoning about the terms being skill-internal turns out to be correct. If the auditor concludes a scan is unnecessary without running the named method, the announcement must read `Scan not performed — <reason>` (e.g., "Scan not performed — changed terms are load-bearing only for skill-creator's internal guidance; no sibling meta-tooling skill documents these conventions"). A scan-not-performed announcement surfaces the possibility that the reasoning is wrong and a sibling DOES use the term; a fabricated "Scanned via grep" announcement buries that possibility under a false method claim that a future reader, trusting the trail, will not re-verify.

  **Extends to user-directed edits on meta-tooling skills**: The sibling-scan discipline also applies when user-directed follow-up implementation modifies a meta-tooling skill itself (e.g., brainstorm, skill-creator, skill-audit) and the edit touches shared terminology, conventions, or tag semantics — even when the edit is not driven by a finding in the current audit report. This includes NEW conventions being introduced by the edit (e.g., a new filename convention, a new tag name, a new enum value) — the scan's purpose is to verify the new convention does not conflict with existing sibling content, regardless of whether the convention was pre-existing or net-new. Before finalizing such an edit, grep sibling meta-tooling skills for references to the changed terminology or convention, and document the scan in the post-implementation summary ("Scanned siblings: <list> — no inconsistencies" or "<list> — adjusted X in Y").
- **Repeated audit shortcut** — If the same skill has been audited *as the target* 2+ times in the current session and the most recent audit found 0 findings, note "Skill stable — no new session evidence since last audit" and skip the full checklist unless the skill was modified between audits. If the skill was modified since the last audit (including by follow-up implementation from a prior audit), treat the next audit as fresh — do not use the shortcut. **Self-audit tie-break**: when the target is skill-audit itself (self-audit), this shortcut DOES apply if the prior self-audit found 0 findings AND no intervening edits to skill-audit — the shortcut phrasing becomes "Skill stable — no new session evidence since last self-audit." Step 3's session-evidence-from-prior-invocations rule governs what evidence a NEW self-audit (one that is NOT covered by the shortcut) draws on; it does not override the shortcut when the shortcut's conditions are met.
