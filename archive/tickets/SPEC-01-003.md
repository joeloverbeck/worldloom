# SPEC-01-003: TypeScript schema types (CF/CH/MH interfaces, node-type + edge-type enums)

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Small
**Engine Changes**: Yes — adds `tools/world-index/src/schema/types.ts` as the single module home for all parser- and CRUD-facing shapes. No existing symbols are renamed or removed.
**Deps**: SPEC-01-001 (package scaffold)

## Problem

Every downstream ticket — Pass 2 YAML validation, Pass 3 edge emission, Pass 4 canonical hashing, nodes/edges CRUD, CLI inspect output — needs a shared TypeScript definition of:

- The 19 `node_type` values (from SPEC-01 §Node types) as a string-literal union + runtime-iterable list.
- The 13 `edge_type` values (from SPEC-01 §Edge types, post-reassessment) grouped by provenance (YAML-field-derived / attribution-comment-derived / entity-virtual) as a union.
- The `CanonFactRecord`, `ChangeLogEntry`, and `ModificationHistoryEntry` interfaces mirroring FOUNDATIONS.md §Canon Fact Record Schema, shaped to tolerate SPEC-09's future conditionally-mandatory `exception_governance` / `action_space` additions without rework.
- The `Node`, `Edge`, `EntityMention`, `FileVersion`, `AnchorChecksum`, `ValidationResult` row shapes matching the SQLite schema from SPEC-01-002.

Without this module, each parser pass would ad-hoc its own copy of these types and drift almost immediately. FOUNDATIONS §Canon Fact Record Schema is the authoritative source — silent field drops at this layer are Rule-6 risks.

## Architecture Check

1. Single-file type module (`src/schema/types.ts`) centralizes every shape a parser, CRUD, or hash path needs. Alternative — scattered per-module types — drifts quickly and undermines the FOUNDATIONS/template citations. No backwards-compatibility aliasing.
2. String-literal unions + exported `readonly`-tuple runtime lists (e.g., `NODE_TYPES: readonly NodeType[]`) give both compile-time exhaustiveness and runtime iteration — needed by Pass 3 ID-regex construction and by SPEC-01-008's `inspect` command.
3. Interfaces mirror the YAML schema 1:1, never collapsing fields into a "normalized" alternate form. Canonical-form hashing (SPEC-01-004) re-serializes with key-sorting; it doesn't need a special AST shape. This avoids a dual-representation failure mode where the interface drifts from the YAML.
