# SPEC-01-012: Preserve `CH-*` identity for schema-incomplete change-log entries without hiding schema drift

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — `tools/world-index/src/parse/yaml.ts` now salvages `CH-*` ids from schema-incomplete change-log blocks, and world-index tests cover both the parser seam and the live animalia regression.
**Deps**: `archive/tickets/SPEC-01-005.md`, `archive/tickets/SPEC-01-006.md`, `archive/tickets/SPEC-01-009.md`, `docs/FOUNDATIONS.md`, `specs/SPEC-04-validator-framework.md`

## Problem

`world-index build animalia` currently leaves `25` unresolved `modified_by` edges pointing at `CH-0010`, `CH-0013`, `CH-0014`, and `CH-0015` even though those `change_id` values are present in `worlds/animalia/CANON_LEDGER.md`. The direct cause is in `tools/world-index/src/parse/yaml.ts`: if a change-log block is missing a required field such as `summary` or `reason`, `narrowToChangeLogEntry()` returns `null`, and the node falls back to a synthetic `__unknown__...yaml:line-line` id. That couples graph identity to full schema completeness, which breaks Rule 6 traceability and cross-file linking.

## Assumption Reassessment (2026-04-22)

1. In `tools/world-index/src/parse/yaml.ts`, the `change_log` branch currently assigns `nodeId: narrowed?.change_id ?? syntheticNodeId(...)`. A parsed block with a real `change_id` but one missing required field still gets a synthetic node id because `narrowed` is `null`.
2. Live `validation_results` for animalia show `3` `missing_required_field:reason` warnings and `1` `missing_required_field:summary` warning on `CANON_LEDGER.md`, and the current `change_log_entry` node set in the DB contains four synthetic fallback ids instead of `CH-0010`, `CH-0013`, `CH-0014`, and `CH-0015`.
3. `worlds/animalia/CANON_LEDGER.md` contains literal `change_id: CH-0010` at line `7548`, `CH-0013` at `7862`, `CH-0014` at `8013`, and `CH-0015` at `8147`, and many `modification_history[].change_id` references point at those ids. The identity data exists in source; the indexer is dropping it.
4. FOUNDATIONS Rule 6 (`No Silent Retcons`) requires canon changes to remain logged and traceable. Missing `summary` or `reason` is still a schema problem, but it should not erase the change id from the graph if the block parsed and the identifier is present.
5. Cross-artifact boundary under audit: Change Log Entry schema in `tools/world-index/src/schema/types.ts`, YAML parsing in `parse/yaml.ts`, and `modified_by` edge extraction in `parse/semantic.ts`.
6. Mismatch + correction: the current implementation treats "schema-incomplete" as if it were "identifier-unknown." This ticket should preserve the declared `change_id` while keeping the schema warning explicit.
7. Verification command drift: the live CLI resolves the world root from `process.cwd()`, so package-local verification must either invoke the exported `build()` path on a temp copy or run `node tools/world-index/dist/src/cli.js ...` from the repo root. The drafted `cd tools/world-index && node dist/src/cli.js build animalia` command is not truthful as written.

## Architecture Check

1. The clean approach is to salvage only the primary identifier from a parsed change-log block when it is present and syntactically valid, while still refusing to narrow the whole object to `ChangeLogEntry` until required fields are complete. That preserves graph identity without laundering schema drift.
2. No backwards-compatibility shim should create duplicate nodes under both synthetic and `CH-*` ids. The end state is one node per change-log block.

## Verification Layers

3. `modification_history[].change_id` references to those entries resolve after build. -> integration test + readonly DB query
4. A block with no `change_id` still falls back to a synthetic id. -> edge-case YAML test

## What to Change

### 1. Separate identifier salvage from schema narrowing

Teach the change-log branch in `extractYamlNodes()` to extract a declared `change_id` from the parsed object even when `narrowToChangeLogEntry()` fails, as long as the id is a non-empty string and matches the structured-id contract.

### 2. Keep schema warnings intact

Do not suppress `missing_required_field:summary` or `missing_required_field:reason`. This ticket is about preserving graph identity, not about pretending malformed change-log records are complete.

### 3. Add live regression coverage

Prove the animalia-specific failure mode directly so future parser changes cannot regress the `modified_by` surface again.

## Outcome

Completion date: 2026-04-22.

The YAML parser now treats change-log identity and full schema completeness as separate concerns. In the `change_log` branch, a parsed block with a syntactically valid `change_id` keeps that `CH-*` node id even when `summary` or `reason` is missing and `narrowToChangeLogEntry()` still rejects the object as a complete `ChangeLogEntry`. No duplicate synthetic alias is created; the node identity is recovered directly from the parsed object.

A dedicated change-log missing-field fixture now proves the parser preserves `CH-0099` while still emitting `missing_required_field:reason`, and the animalia integration lane now asserts that `CH-0010`, `CH-0013`, `CH-0014`, and `CH-0015` no longer remain unresolved `modified_by` targets after build.
