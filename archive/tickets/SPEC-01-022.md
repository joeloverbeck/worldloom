# SPEC-01-022: Retune `named_entity` extraction against embedded workflow-checkpoint prose in whole-file records

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — `tools/world-index` heuristic entity extraction, stoplist coverage, and rebuilt-`animalia` regression coverage for whole-file record bodies.
**Deps**: `archive/tickets/SPEC-01-020.md`, `archive/tickets/SPEC-01-021.md`, `docs/FOUNDATIONS.md`, `archive/specs/SPEC-01-world-index.md`

## Problem

Fresh rebuild audit on 2026-04-22 shows `worlds/animalia/_index/world.db` is structurally clean but still semantically noisy on the `named_entity` surface. The rebuilt DB has `validation_results = 0`, unresolved edges = `0`, dangling edge/mention references = `0`, and `verify animalia` exits cleanly, yet it still persists workflow/checkpoint phrases from whole-file records as first-class entities.

Reproduced false positives include:

- `Canon Safety Check Trace` sourced from `diegetic-artifacts/a-season-on-the-circuit.md`, `diegetic-artifacts/after-action-report-harrowgate-contract.md`, and `proposals/batches/BATCH-0001.md`
- `Truth Check` sourced from the `World-Truth Check` / `Narrator-Truth Check` subsections inside both diegetic artifacts
- `Primary Rule` sourced from proposal-card risk prose such as `Primary Rule-3 risk` in `proposals/PR-0010-clergy-witness-procedural-recital-cannibal-case.md`
- `Per Phase` sourced from workflow/meta prose in `proposals/PR-0015-corner-share-contractor-auxiliary-compensation.md`, `proposals/PR-0017-drylander-short-poetry-stranger-hospitality.md`, and `OPEN_QUESTIONS.md`

Those rows reproduce after a fresh rebuild, so they are not stale checked-in artifact residue. They are semantic noise in the persisted graph that downstream retrieval/audit consumers would have to subtract manually.

## Assumption Reassessment (2026-04-22)

1. `tools/world-index/src/commands/shared.ts` still includes `character_record`, `diegetic_artifact_record`, `proposal_card`, `proposal_batch`, `character_proposal_card`, `character_proposal_batch`, and `retcon_proposal_card` inside `ENTITY_SOURCE_NODE_TYPES`, so whole-file records remain eligible heuristic entity sources.
2. `tools/world-index/src/parse/prose.ts` emits those record types as whole-file nodes with `body: lines.join("\n")`, so internal workflow/checkpoint prose such as `## Canon Safety Check Trace`, `World-Truth Check`, `Narrator-Truth Check`, and `Primary Rule-3 risk` is scanned as part of the same entity-extraction surface as real world-facing prose.
3. `tools/world-index/src/parse/entities.ts` still applies `CAPITALIZED_MULTIWORD_REGEX` over those bodies and only blocks candidates through `FRAGMENT_LEAD_WORDS` plus `isStoplistedEntityCandidate()`. The currently reproduced rows show that the earlier stoplist work does not yet cover embedded canon-workflow/checkpoint language inside whole-file records.
4. Shared boundary under audit: whole-file prose-node construction in `tools/world-index/src/parse/prose.ts`, source-node selection in `tools/world-index/src/commands/shared.ts`, heuristic candidate extraction/filtering in `tools/world-index/src/parse/entities.ts` and `tools/world-index/src/parse/stoplist.ts`, and the persisted `named_entity` / `entity_mentions` rows in SQLite.
5. `docs/FOUNDATIONS.md` requires canon facts to reinforce the world model rather than machine-ingesting workflow scaffolding as if it were world ontology. Persisting `Canon Safety Check Trace`, `Truth Check`, or `Primary Rule` as entities weakens that contract instead of clarifying it.
6. Archived `SPEC-01-020` and `SPEC-01-021` already removed earlier waves of workflow-label and fragment noise (`Mystery Reserve`, `Continuity Archivist`, `Distribution Discipline`, `World Laundering`, `In Brinewick`, `An Ash`, and related rows). This audit reproduces a narrower adjacent seam that survived those fixes rather than reopening the same ownership class.
7. The current file-coverage gap is not part of this seam. `file_versions` misses only the five `INDEX.md` files, and `archive/tickets/SPEC-01-007.md` explicitly excludes `INDEX.md` from the indexable include set.
8. No active ticket currently owns this exact class. `tickets/` contains no other `SPEC-01-*` ticket, and grep across `tickets/` plus `archive/tickets/` found no existing owner for `Canon Safety Check Trace`, `Truth Check`, `Primary Rule`, or `Per Phase`.
9. Mismatch + correction: the remaining issue is semantic, not structural. The truthful follow-up is another bounded `named_entity` extractor retune rather than a schema/integrity ticket.
10. Mismatch + correction: `tools/world-index/src/parse/prose.ts` did not need a whole-file/body-shape change. The live extractor seam was still fixable inside `tools/world-index/src/parse/entities.ts` plus `tools/world-index/src/parse/stoplist.ts`, so the ticket narrowed away from a `prose.ts` edit before implementation.

## Architecture Check

1. The clean fix is to keep the canonical `named_entity` surface trustworthy by stripping or ignoring embedded workflow/checkpoint prose before heuristic scanning, or by stoplisting the surviving checkpoint phrase class at the extractor boundary. That is cleaner than teaching downstream consumers to maintain another layer of semantic exclusions.
2. No backwards-compatibility aliasing or parallel "filtered entities" surface should be introduced. The persisted `named_entity` rows themselves should stop containing workflow/checkpoint labels.

## Verification Layers

1. Embedded workflow/checkpoint labels from whole-file records no longer persist as `named_entity` rows after a fresh `animalia` rebuild. -> readonly DB query for `Canon Safety Check Trace`, `Truth Check`, `Primary Rule`, and `Per Phase`
2. Legitimate retained entities in the same source families remain present after the retune. -> readonly DB query for `Vespera Nightwhisper`, `Melissa Threadscar`, `Copper Weir`, and `Charter Hall`
3. The fix stays semantic-only; the rebuilt DB remains structurally clean. -> `node tools/world-index/dist/src/cli.js build animalia`, `node tools/world-index/dist/src/cli.js verify animalia`, plus readonly unresolved/dangling queries
4. Whole-file record parsing remains intact; only heuristic entity extraction over embedded workflow/checkpoint prose changes. -> targeted unit/integration coverage on `tools/world-index`

## What to Change

### 1. Retune the extractor boundary for embedded checkpoint prose

Keep whole-file record bodies eligible for heuristic scanning, but block the surviving checkpoint/meta phrase class directly at the extractor boundary so real record prose remains indexable.

### 2. Retune the surviving checkpoint phrase class

Prevent persisted entity rows for the reproduced checkpoint/meta phrases, including `Canon Safety Check Trace`, `Truth Check`, `Primary Rule`, and `Per Phase`, without regressing earlier retained positives from diegetic artifacts, character files, or proposal cards.

### 3. Add rebuilt-`animalia` regression coverage

Encode the audited false positives directly in package tests so future rebuilds cannot silently repollute the semantic layer with checkpoint-language entities.
