# SPEC-01-014: Resolve `required_world_updates` to stable domain-file nodes

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Large
**Engine Changes**: Yes — this ticket extends the world-index node contract so mandatory/root world files have stable node identities that `required_world_update` edges can target.
**Deps**: `archive/tickets/SPEC-01-006.md`, `archive/tickets/SPEC-01-008.md`, `archive/tickets/SPEC-01-009.md`, `docs/FOUNDATIONS.md`, `specs/SPEC-02-retrieval-mcp-server.md`, `specs/SPEC-04-validator-framework.md`

## Problem

`world-index build animalia` currently leaves `180` unresolved `required_world_update` edges targeting filenames like `INSTITUTIONS.md`, `EVERYDAY_LIFE.md`, and `ECONOMY_AND_RESOURCES.md`. This is an architectural gap, not bad source data: FOUNDATIONS defines `required_world_updates` as filenames, `SPEC-02` expects downstream "domain-file nodes" for impacted-fragment queries, but the live index has no stable node identity for top-level domain files and `resolveUnresolvedEdges()` can only resolve exact `node_id` matches.

## Assumption Reassessment (2026-04-22)

1. FOUNDATIONS `§Canon Fact Record Schema` explicitly defines `required_world_updates` as an array of filenames such as `INSTITUTIONS.md`, `ECONOMY_AND_RESOURCES.md`, and `EVERYDAY_LIFE.md`.
2. `tools/world-index/src/parse/semantic.ts` currently calls `resolveWorldUpdateTarget(target, proseNodes)`, but `proseNodes` in `parseWorldFile()` are only the current file's nodes. For a CF in `CANON_LEDGER.md`, no current-file prose node can ever represent `EVERYDAY_LIFE.md`.
3. `tools/world-index/src/index/edges.ts` later resolves unresolved edges only where `edges.target_unresolved_ref = nodes.node_id`. Since there are no node ids equal to bare filenames like `INSTITUTIONS.md`, those `required_world_update` edges remain unresolved forever.
4. `specs/SPEC-02-retrieval-mcp-server.md` states that `find_impacted_fragments(node_ids)` computes downstream impact via `required_world_update + mentions_entity` edges and expects "domain-file nodes likely affected." The current graph shape cannot satisfy that contract truthfully.
5. `specs/SPEC-04-validator-framework.md` defines `cross_file_reference` as requiring every id referenced in `required_world_updates` to resolve to an indexed node. The current node model makes that impossible for valid FOUNDATIONS-authored filenames.
6. Cross-artifact boundary under audit: Canon Fact Record `required_world_updates`, world-index node identity for top-level domain files, `required_world_update` edges, and retrieval/validator consumers.
7. The drafted command `cd tools/world-index && node dist/src/cli.js build animalia` is not a truthful proof command because the CLI resolves worlds from `process.cwd()`. The honest direct CLI form is repo-rooted (`node tools/world-index/dist/src/cli.js ...`) or package-local tests that copy the world into a temp repo root.
8. The checked-in `worlds/animalia/_index/` is not a trustworthy proof surface for this ticket; a direct root-level query hit stale-schema fallout (`no such table: file_versions`). The honest animalia proof remains the integration test's temp-copy rebuild plus readonly DB assertions against that rebuilt copy.
9. Mismatch + correction: unresolved `required_world_update` edges are not an animalia content bug. They are evidence that the current node model is missing the domain-file target surface the specs already assumed.

## Architecture Check

1. The clean solution is to add an explicit stable `domain_file` node surface for top-level world files, then resolve `required_world_update` edges to those nodes by exact relative-path contract. That is cleaner than overloading a random first section node or continuing to treat bare filenames as unresolved forever.
2. No backwards-compatibility aliasing should preserve the broken "filename stored forever in `target_unresolved_ref`" contract. The point of this ticket is to replace that gap with a real target node.

## Verification Layers

1. Every `required_world_update` filename in a fixture world resolves to a stable top-level `domain_file` node after build. -> targeted command test
2. Animalia temp-copy build leaves `0` unresolved `required_world_update` edges. -> integration test + readonly DB query
3. `find_impacted_fragments`'s assumed graph substrate becomes truthful because the file-target node surface exists. -> spec alignment check + grep-proof
4. A filename not represented by a real world file still fails truthfully as unresolved. -> edge-case command test

## What to Change

### 1. Introduce stable top-level domain-file nodes

Extend the node contract so mandatory/root world files such as `WORLD_KERNEL.md`, `INVARIANTS.md`, `EVERYDAY_LIFE.md`, and `INSTITUTIONS.md` each emit one stable `domain_file` node with an exact relative-path-based identity that other graph edges can target.

### 2. Resolve `required_world_update` against that new node surface

Replace the current same-file-only heuristic in `resolveWorldUpdateTarget()` with a lookup that targets the stable domain-file node identity defined above.

### 3. Add end-to-end proof for retrieval-facing behavior

Protect the fix with both fixture-world coverage and the live `animalia` corpus so the `required_world_update` substrate remains compatible with downstream retrieval and validation work.
