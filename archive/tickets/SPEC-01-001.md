# SPEC-01-001: Package scaffold & toolchain for tools/world-index/

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Small
**Engine Changes**: Yes — new package `tools/world-index/`; no impact on existing skills, tools, hooks, or docs.
**Deps**: None (Phase 0 scaffold at `tools/world-index/README.md` is the starting point; `.gitignore` already excludes `tools/*/dist/` and `tools/*/node_modules/`)

## Problem

SPEC-01 requires `tools/world-index/` as a standalone TypeScript package delivering a `world-index` CLI. Today the directory contains only `README.md` (Phase 0 scaffold). Nothing else in this spec can land until the package has a buildable shell: `package.json` declaring concrete npm dependencies, `tsconfig.json` compiling to `dist/`, the `src/` directory layout prescribed by the spec, and a minimal `src/cli.ts` entry that prints `--version` / `--help`. Downstream tickets (SQLite schema, parser passes, CRUD, CLI commands) all depend on this shell.

## Assumption Reassessment (2026-04-22)

1. `tools/world-index/` exists as a Phase 0 scaffold with only `README.md`; sibling directories `tools/world-mcp/`, `tools/patch-engine/`, `tools/validators/`, and `tools/hooks/` are also README-only, so there is no in-repo package template to copy.
2. SPEC-01 §Package location mandates the exact `src/` layout (`cli.ts`, `commands/`, `parse/`, `schema/`, `index/`, `hash/`, `tests/fixtures/`, `tests/integration/`). SPEC-01 §Package location also enumerates npm deps: `better-sqlite3`, `unified`, `remark-parse`, `remark-gfm`, `yaml`. The corresponding Phase 0 `README.md` at `tools/world-index/README.md` lists the same five deps.
3. Cross-artifact boundary: the package must be importable by `tools/world-mcp/`, `tools/patch-engine/`, `tools/validators/` as a **black-box producer** of `worlds/<slug>/_index/world.db`. Per SPEC-01 §Public contract, this package does NOT export a programmatic query library — sibling packages open `world.db` directly via their own `better-sqlite3` connections. Package's `package.json` `main`/`types` fields should not advertise a public API beyond the CLI binary.
4. Root `.gitignore` already excludes `tools/*/dist/` and `tools/*/node_modules/`; no ignore changes are needed for this scaffold.
5. Mismatch corrected before implementation: the authored `tsconfig.json` shape (`rootDir: "./src"` plus `include: ["src/**/*", "tests/**/*"]`) is not a truthful compile contract for a top-level `tests/` tree. The live implementation uses `rootDir: "."` so both `src/` and `tests/` compile into `dist/`.

## Architecture Check

1. Choosing `better-sqlite3` (synchronous) over `sqlite3` (async) aligns with SPEC-01's determinism goal — synchronous I/O keeps parse-then-write ordering explicit and makes content-hash reproducibility trivial. Async SQLite would require promise chains that complicate hash-stable iteration order.
2. Keeping the package a **pure producer** with no library export surface preserves the public contract boundary declared in SPEC-01: `world.db` is the contract, not a TypeScript API. No backwards-compatibility aliasing or shims.
3. Pinning concrete npm-package versions in `package.json` (rather than `^` ranges) for the five deps protects cross-platform content-hash stability (`remark` AST shape drift between minor versions could move byte offsets).

## Verification Layers

1. Package scaffolds on a fresh machine → `npm install` succeeds with the pinned deps; `npm run build` produces a runnable compiled CLI under `dist/`.
2. `world-index --version` / `world-index --help` exit 0 → CLI bin wiring correct.
3. `tsc --noEmit` passes → TypeScript configuration is coherent before any parser code is added.
4. Grep-proof that no sibling `tools/*` package imports from `tools/world-index/src/` directly → public-contract invariant (file-is-contract) holds from day one.
