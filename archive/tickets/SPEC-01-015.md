# SPEC-01-015: Repair live animalia schema-drift warnings surfaced by `world-index build`

**Status**: COMPLETED
**Priority**: MEDIUM
**Effort**: Medium
**Engine Changes**: None — this is a corpus/data-contract repair ticket against `worlds/animalia`, proven with the existing world-index build surface.
**Deps**: `docs/FOUNDATIONS.md`, `specs/SPEC-04-validator-framework.md`, `archive/tickets/SPEC-01-012.md`

## Problem

Even after the indexer seams above are fixed, the live `animalia` corpus still has real schema-drift warnings that should not be normalized away. Current `world-index build animalia` records `3` `missing_required_field:reason` warnings, `1` `missing_required_field:summary` warning, and `1` `unexpected_yaml_section` warning. FOUNDATIONS and the validator spec both treat these as real contract violations, not acceptable steady-state noise.

## Assumption Reassessment (2026-04-22)

1. `docs/FOUNDATIONS.md` requires complete Change Log Entries and treats canon-change traceability as non-negotiable under Rule 6. Missing `summary` or `reason` on a real `CH-*` entry is therefore a real content defect, even if the indexer learns to preserve the `change_id`.
2. `specs/SPEC-04-validator-framework.md` defines `yaml_parse_integrity` as checking that every fenced YAML block parses and has required fields present per `CanonFactRecord` / `ChangeLogEntry` schemas.
3. Temp-copy build evidence on 2026-04-22 confirmed the exact current warning set: `CH-0010`, `CH-0013`, `CH-0014`, and `CH-0015` in `worlds/animalia/CANON_LEDGER.md` were producing the live `missing_required_field:*` warnings, and `worlds/animalia/adjudications/PA-0016-accept_with_required_updates.md` produced the sole `unexpected_yaml_section` warning.
4. Mismatch + correction: the adjudication warning is not caused by a parser-recognized adjudication-discovery section. In `tools/world-index/src/parse/yaml.ts`, any fenced YAML block outside `## Canon Fact Records` or `## Change Log` is classified as `other` and flagged as `unexpected_yaml_section`, so this ticket's truthful fix is to rewrite the `PA-0016` discovery metadata out of fenced YAML rather than "move it under a canonical adjudication section."
5. Cross-artifact boundary under audit: `worlds/animalia/CANON_LEDGER.md`, `worlds/animalia/adjudications/PA-0016-accept_with_required_updates.md`, FOUNDATIONS change-log schema, and the validator-facing `yaml_parse_integrity` surface.
6. Command-shape correction: the truthful proof surface for this ticket is a temp-copy build via the exported `build()` function plus readonly SQLite queries against the copied `_index/world.db`, not `cd tools/world-index && node dist/src/cli.js build animalia`, because the CLI resolves the world root from `process.cwd()` and the checked-in `worlds/animalia/_index/` tree is not a reliable acceptance surface.
7. Mismatch + correction: the indexer should not be used as an excuse to tolerate malformed canon records. This ticket owns the corpus repair after the graph-identity ticket makes the failures easier to inspect.

## Architecture Check

1. The clean solution is to repair the authored data so it satisfies the documented schema, rather than weakening the parser or validator to accept steady-state malformed canon records.
2. No backwards-compatibility shims or alternative change-log/adjudication schemas should be introduced.
