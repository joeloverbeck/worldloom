# SPEC-01-016: Make `world-index build` a truthful clean-rebuild path

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Small
**Engine Changes**: Yes — `tools/world-index` rebuild/bootstrap behavior and its command coverage.
**Deps**: `archive/specs/SPEC-01-world-index.md`, `archive/tickets/SPEC-01-012.md`, `archive/tickets/SPEC-01-015.md`

## Problem

The current full-build path is not a reliable "delete and rebuild from source" recovery surface. In `tools/world-index/src/commands/shared.ts`, `buildWorldIndex()` deletes `world.db` but leaves `_index/index_version.txt` in place. `openIndex()` then sees a recorded schema version and skips the initial migration, which leaves a fresh SQLite file without the required tables. A subsequent query crashes with `no such table: file_versions` / `no such table: nodes` instead of performing a truthful clean rebuild.

## Assumption Reassessment (2026-04-22)

1. `archive/specs/SPEC-01-world-index.md` defines deleting `_index/world.db` and running `world-index build` as the canonical recovery path for a regenerable per-world index.
2. In the live implementation, `tools/world-index/src/commands/shared.ts` calls `deleteFile(dbPath)` before `openIndex(worldRoot, worldSlug)`, but `tools/world-index/src/index/open.ts` only runs `001_initial.sql` when `index_version.txt` is missing, not when `world.db` is missing.
3. Live repro on 2026-04-22 from the repo root: `node tools/world-index/dist/src/cli.js build animalia` after a stale version-marker state produced `no such table: file_versions`, and follow-up `stats` / `verify` calls failed similarly until the version marker was moved aside.
4. Shared boundary under audit: `_index/world.db` bootstrap state, `_index/index_version.txt` schema-version state, and the exported/CLI `build` command contract.
5. Mismatch + correction: the current code treats "schema version marker exists" as equivalent to "database schema exists." That assumption is false once `build` deletes only the database file.
6. Adjacent issue classification: the CLI's dependence on `process.cwd()` for repo-root resolution is a separate ergonomics defect and should not be silently folded into this ticket unless the chosen rebuild fix naturally centralizes root discovery.
7. Reassessment narrowed the owned code delta to `tools/world-index/src/index/open.ts`. `tools/world-index/src/commands/shared.ts` correctly performs the full-build database deletion; the bug is the bootstrap path that skips migration when only the version marker survives.

## Architecture Check

1. The clean solution is to make bootstrap state derive from the actual database plus version-marker pair, not from the version marker alone. A full rebuild should leave one canonical path to a migrated database, not a half-reset state.
2. No backwards-compatibility shim should preserve the broken "delete DB only, trust stale version marker" behavior.

## Verification Layers

1. A full rebuild from a state where `_index/index_version.txt` exists but `world.db` is absent recreates the schema instead of crashing. -> targeted tool command
2. The rebuilt database contains the canonical schema tables (`nodes`, `edges`, `file_versions`, `validation_results`, FTS tables). -> readonly DB query
3. `stats` and `verify` succeed immediately after that rebuild on a temp-copied `animalia` world. -> targeted tool command
4. The clean-rebuild contract remains aligned with the SPEC-01 "regenerable from markdown" design. -> FOUNDATIONS/spec alignment check

## What to Change

### 1. Fix clean-rebuild bootstrap semantics

Update `tools/world-index/src/index/open.ts` so a full rebuild cannot leave a fresh database file paired with a stale schema marker and no migrated tables.

### 2. Add explicit regression coverage

Add a test that simulates the stale-marker / missing-database state and proves `build` recreates a usable index.

### 3. Keep the proof surface truthful

Use repo-root or exported-command verification paths that match the live CLI/root-resolution behavior instead of papering over the current failure mode.

## Outcome

Completion date: 2026-04-22.
Outcome amended: 2026-04-22.

`openIndex()` now treats the actual bootstrap pair as authoritative: if `world.db` is absent, it runs the initial migration and rewrites `_index/index_version.txt` even when a stale version marker is already present. That removes the half-reset state where `build` created a fresh SQLite file with no schema because only the version marker survived.

The animalia integration suite now includes a regression case that temp-copies the world, writes only `_index/index_version.txt`, runs `build`, confirms the canonical schema objects exist (`nodes`, `edges`, `file_versions`, `validation_results`, `fts_nodes`), and then proves `stats` and `verify` succeed immediately against that rebuilt copy.

Post-ticket review briefly exposed a brittle assertion in that new regression: the test matched `/nodes?:/i` while `stats()` prints `Node counts:`. The final closeout corrected the assertion to the actual header and reran the proof lane cleanly before archival.

## Verification Result

2. `cd tools/world-index && node --test dist/tests/integration/build-animalia.test.js` — passed on the final sequential rerun after correcting the stale-marker regression's `stats` header assertion to match the live `Node counts:` output.
