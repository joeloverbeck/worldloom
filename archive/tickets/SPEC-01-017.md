# SPEC-01-017: Normalize virtual `named_entity` node provenance to the node schema contract

**Status**: COMPLETED
**Priority**: MEDIUM
**Effort**: Small
**Engine Changes**: Yes — `tools/world-index` synthetic entity-node construction plus tests/audit coverage.
**Deps**: `archive/specs/SPEC-01-world-index.md`, `archive/tickets/SPEC-01-010-entity-heading-seam.md`

## Problem

The production `animalia` index currently writes malformed provenance metadata for every synthetic `named_entity` node. All `614` `named_entity` rows use an absolute `file_path` (`/home/.../worlds/animalia/ONTOLOGY.md`) while the rest of the index uses world-relative paths, and those same rows store `line_start = line_end = byte_start = byte_end = 0`. That breaks the database's own normalization pattern and makes generic source-range audits flag every synthetic entity row as invalid.

## Assumption Reassessment (2026-04-22)

1. `archive/specs/SPEC-01-world-index.md` defines `nodes.file_path`, `byte_start`, `byte_end`, `line_start`, and `line_end` as part of the public node schema consumed directly from `world.db`.
2. The live constructor in `tools/world-index/src/parse/entities.ts` used `loadOntologyRegistry()`'s absolute filesystem path and stored `line_start = line_end = byte_start = byte_end = 0` for every synthetic entity node, while other persisted nodes use world-relative `file_path` values and nonzero line spans.
3. Readonly audit queries against `worlds/animalia/_index/world.db` on 2026-04-22 showed `614` malformed rows, and every one of them was a `named_entity` row with an absolute path under the local checkout plus all-zero source metadata.
4. Shared boundary under audit: `nodes` table schema, synthetic `named_entity` rows, and downstream consumers that join or filter on `file_path` and source spans.
5. Mismatch + correction: synthetic rows do not need per-mention source coordinates, but they do need schema-consistent provenance. This ticket normalizes them to the world-relative ontology registry file with a documented sentinel span (`ONTOLOGY.md`, line `1`, byte `0`) instead of malformed absolute-path/all-zero metadata.

## Architecture Check

1. The clean solution is to give virtual nodes a consistent, explicit provenance convention that still satisfies the `nodes` schema contract, rather than expecting consumers to special-case absolute paths and zero spans.
2. No backwards-compatibility aliasing should preserve both malformed and normalized provenance shapes.

## Verification Layers

3. Existing entity-node consumers and tests still resolve `mentions_entity` edges correctly after provenance normalization. -> targeted tool command

## What to Change

### 1. Normalize synthetic node provenance

Update synthetic `named_entity` construction so `file_path` is stored in the same relative form as other nodes, and use a single documented sentinel source-location convention for virtual nodes instead of malformed parser output.

### 2. Add regression coverage

Add test assertions that synthetic entity nodes emit relative ontology paths and the agreed sentinel span.

### 3. Preserve consumer clarity

If a special virtual-node convention is needed, document it in the tests/spec-adjacent comments instead of leaving it implicit in malformed values.
