## Problem

Every prior ticket has produced library modules inside the package. This ticket exposes them as five CLI commands that downstream Phase 1 work depends on: `build <world-slug>` (full rebuild), `sync <world-slug>` (incremental), `inspect <node-id>` (single-node dump), `stats <world-slug>` (counts), `verify <world-slug>` (re-hash + drift flag). Exit codes 0/1/2/3/4 match SPEC-01 §CLI contract. The incremental sync algorithm from SPEC-01 §Incremental sync algorithm composes the parser passes, CRUD module, and enumeration inventory into a coherent invocation flow.

## Assumption Reassessment (2026-04-22)

1. SPEC-01 §CLI contract enumerates the five commands + `--version` / `--help` + exit codes `0` success, `1` generic failure, `2` invalid world slug, `3` missing mandatory file, `4` parse failure above threshold. SPEC-01 §Incremental sync algorithm (post-reassessment) now references §File-enumeration inventory explicitly and notes FTS5 coherence via triggers.
2. SPEC-01 §Public contract (post-reassessment): this CLI is the canonical producer entry point. Sibling tooling reads `world.db` directly via their own `better-sqlite3` connections. The CLI never hands off a long-lived database connection to callers.
3. Cross-artifact boundary: `world-index sync <slug>` is invoked by Hook 5 (SPEC-05 Part B) after every successful write-path patch-engine commit. This ticket exposes the CLI; Hook 5 wires the invocation. `world-index build <slug>` is invoked by SPEC-08 Phase 1 bootstrap.
4. Incremental sync threshold logic: the "parse failure above threshold" exit code 4 fires when the ratio of `yaml_parse_integrity` fail-severity rows to total YAML blocks exceeds 10%. This is a CLI-level decision that consumes the `validation_results` counts this sync run produced — it doesn't re-query the stored table.
5. FOUNDATIONS principle under audit: **Rule 6 (No Silent Retcons)** — `verify` detects post-edit drift by re-hashing every node and comparing against stored `content_hash`. Surfacing drift is the structural preservation of Rule 6 at the retrieval layer; SPEC-04's validator surface consumes these flags.

## Architecture Check

1. Each command is a thin composition in its own file under `src/commands/`. `cli.ts` is pure argument parsing + dispatch. This keeps the CLI surface testable (each command callable as a library function from tests) and the argument parsing centralized. Alternative — fat-cli dispatch with inline command bodies — was rejected for testability.
2. The incremental-sync algorithm implementation is a linear composition of `enumerate` → `compareHashes(file_versions)` → `deleteNodesByFile` → `parseMarkdown` → `extractYamlNodes` → `extractSemanticEdges` → `extractEntities` → `insertNodes/Edges/Mentions/Hashes` → `resolveUnresolvedEdges` → `shouldRebuildFts ? rebuildFtsIndex : skip` → `upsertFileVersion`. No state is carried across files except the DB handle — failures can abort a single file without corrupting earlier-file progress because each file's writes happen inside `db.transaction`.
3. Exit codes mapped via a single `cliErrorHandler(err: unknown): number` function (tests can assert return values). No backwards-compatibility shims.

## Verification Layers

1. `world-index build fixture-world` against a seeded fixture produces a `world.db` whose node count matches expected; re-running produces identical `content_hash` on every node (determinism check). → end-to-end smoke test (SPEC-01-009 owns the animalia scale check; this ticket tests a tiny fixture).
2. `world-index sync fixture-world` after touching one file's bytes re-parses only that file (measured via `file_versions` before/after + test-instrumented enumeration count). → incremental selectivity.
3. `world-index inspect CF-0001` against a seeded DB dumps the node's body + edges + entity mentions as JSON + exits 0. → inspect output shape.
4. `world-index stats fixture-world` prints per-node-type counts matching `SELECT node_type, COUNT(*) FROM nodes GROUP BY node_type` → stats consistency.
5. `world-index verify fixture-world` after manually editing one line in a domain file flags the edited node with non-zero exit → drift detection.
6. `world-index build <nonexistent-slug>` exits 2; `world-index build` (missing slug) exits 1 with usage; `world-index build --help` exits 0 → exit-code contract.
7. FOUNDATIONS alignment: `verify` emits a `validation_results` drift row for every stale hash → grep-proof that this module is the structural enforcement site for Rule 6 drift detection.

## What to Change

### 1. `tools/world-index/src/cli.ts` (replacing SPEC-01-001 stub)

Argument parsing via `node:util.parseArgs` (no `commander` or other third-party CLI library — keeps dep surface small). Route `argv[2]` to one of: `build`, `sync`, `inspect`, `stats`, `verify`, `--version`, `--help`. Unknown subcommand → print usage + exit 1.

### 2. `tools/world-index/src/commands/build.ts`

Exports `async function build(worldRoot: string, worldSlug: string): Promise<number>` (returns exit code):

1. Resolve world root. If `<worldRoot>/worlds/<worldSlug>/` is not a directory, return `2` (invalid world slug).
2. Delete `<worldRoot>/worlds/<worldSlug>/_index/world.db` if present (full rebuild discards prior state).
3. Call `openIndex` — creates fresh DB + migration + `index_version.txt`.
4. `enumerate(worldRoot, worldSlug)` — warn-log any `unexpected` paths as `validation_results` rows.
5. For each indexable file, run the parser pipeline (parseMarkdown → extractYamlNodes → extractSemanticEdges → extractEntities → anchorChecksum compute), collect all rows, batch-insert via CRUD.
6. `resolveUnresolvedEdges(db)` once all files are indexed.
7. `rebuildFtsIndex(db)` (full build always triggers the rebuild optimization).
8. Compute parse-failure ratio; if `> 0.1`, return `4`; otherwise `0`.

### 3. `tools/world-index/src/commands/sync.ts`

Exports `async function sync(worldRoot, worldSlug): Promise<number>`:

Algorithm per SPEC-01 §Incremental sync algorithm:

1. `openIndex` — if DB doesn't exist OR version mismatches, return `1` with message instructing to run `build` instead (not `0` / auto-rebuild — stays explicit per Rule 6 discipline).
2. `enumerate` current filesystem → `indexable` + `unexpected`.
3. For each indexable file: compute its content hash; compare against `file_versions`. If hash matches, skip.
4. For each changed file (or missing from `file_versions`): `deleteNodesByFile` (cascades edges + mentions) → run parser pipeline → batch-insert → `upsertFileVersion`. All writes for one file happen inside one `db.transaction`.
5. After all file-level updates: `resolveUnresolvedEdges`.
6. Compute churn ratio; if `shouldRebuildFts(churnCount, totalNodes)` → `rebuildFtsIndex`.
7. Compute parse-failure ratio; if `> 0.1`, return `4`; else `0`.

### 4. `tools/world-index/src/commands/inspect.ts`

Exports `async function inspect(worldRoot, nodeId): Promise<number>`:

1. Scan `worldRoot/worlds/*/` for the first DB containing a row with `node_id = nodeId`. Alternative — requiring an explicit `<world-slug>` argument — deferred; SPEC-01's CLI contract specifies `inspect <node-id>` without slug. If multiple worlds contain matching ids (shouldn't happen for structured IDs), report ambiguity + exit 1.
2. Dump the node's columns + outgoing edges + incoming edges + entity mentions as pretty-printed JSON to stdout. Return 0.

### 5. `tools/world-index/src/commands/stats.ts`

Exports `async function stats(worldRoot, worldSlug): Promise<number>`:

1. Open DB. Query `SELECT node_type, COUNT(*) FROM nodes GROUP BY node_type ORDER BY node_type`.
2. Query `SELECT file_path, last_indexed_at FROM file_versions ORDER BY file_path` and report file freshness.
3. Print both tables as human-readable text. Return 0.

### 6. `tools/world-index/src/commands/verify.ts`

Exports `async function verify(worldRoot, worldSlug): Promise<number>`:

1. Open DB. For each file in `file_versions`, re-read from disk, re-parse, re-hash each produced node. Compare each computed `content_hash` against the stored `content_hash` in `nodes`.
2. For every mismatch, emit a `validation_results` row with `validator_name='drift_check'`, `severity='fail'`, `node_id`, and the computed vs stored hashes in `message`.
3. If any mismatch was recorded, return 1; else return 0.

### 7. Help text + error handler

`src/cli.ts` help text enumerates every command + exit code from SPEC-01 §CLI contract + §Error model. Generic `cliErrorHandler` maps typed errors from the package (e.g., `SchemaVersionMismatchError`) to exit codes.
