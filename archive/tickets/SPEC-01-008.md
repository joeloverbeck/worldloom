## Problem

`tools/world-index` already had the parser, schema, CRUD, and fixture substrate from the earlier SPEC-01 tickets, but the package still exposed only a stub CLI. Phase-1 consumers need a real producer entry point: `build`, `sync`, `inspect`, `stats`, and `verify`. Reassessment against the live repo also showed a required consequence the draft under-described: the command pipeline could not be truthful without a package-local prose/file-node extractor, because only YAML-backed nodes were implemented.

## Assumption Reassessment (2026-04-22)

1. The live package already contained `enumerate.ts`, `index/open.ts`, CRUD helpers, and Pass 1-3 parser modules. The active gap was not raw package scaffolding; it was command orchestration plus the still-missing prose/file-node ingestion needed to populate non-YAML node types named by `specs/SPEC-01-world-index.md`.
2. `specs/SPEC-01-world-index.md` remains authoritative for the CLI contract: five commands plus `--help` / `--version`, exit codes `0/1/2/3/4`, and the producer-only public contract where sibling tools read `world.db` directly. The landed implementation preserves that producer boundary.
3. Cross-artifact boundary: this ticket owns the CLI producer surface and the incremental sync composition over the already-landed parser/CRUD modules. It does not own MCP wiring, hook wiring, or SPEC-01-009’s animalia-scale integration proof.
4. The drafted "parse failure above threshold" wording assumed fail-severity `yaml_parse_integrity` rows, but the live YAML extractor records syntax failures as `warn`. The landed exit-code `4` logic therefore truthfully uses the current syntax-error count from this run’s YAML parse issues rather than inventing a new severity scheme mid-ticket.
5. FOUNDATIONS principle under audit: **Rule 6 (No Silent Retcons)**. `verify` re-parses indexed files and records `drift_check` failures when stored `content_hash` values no longer match the source markdown. `sync` also keeps rebuilds explicit: a missing or mismatched index does not silently auto-rebuild.
6. Mismatch + correction: the draft `Files to Touch` omitted the required consequence fallout in `tools/world-index/src/parse/prose.ts`, `tools/world-index/src/commands/shared.ts`, and the richer fixture-world markdown needed for an honest end-to-end proof. Those were added because the CLI could not otherwise produce the node types the spec already promised.

## Architecture Check

1. The landed structure keeps `cli.ts` focused on argument parsing, usage text, and exit-code mapping, while each command module exposes a callable function that tests can invoke directly. That is cleaner than a monolithic CLI file and kept the command proof surface small.
2. The incremental command pipeline is centralized in `src/commands/shared.ts`: enumerate files, compare `file_versions`, parse changed files, rewrite per-file rows transactionally, rebuild the global entity state, resolve unresolved edges, and rebuild FTS when needed. This is cleaner than duplicating partial pipeline logic across `build` and `sync`.
3. The required prose/file-node fallout was isolated into `src/parse/prose.ts` instead of being buried inside a command module. That keeps the parsing contract reusable and makes the command layer a composition surface rather than an ad hoc parser.
4. No backwards-compatibility alias paths or duplicate CLI surfaces were introduced.

## Deviations

1. The draft called for `async` command functions and a separate `errors.ts`; the landed implementation kept the commands synchronous because the current pipeline is fully synchronous under `better-sqlite3`, and `cliErrorHandler` plus `SchemaVersionMismatchError` in `index/open.ts` were sufficient.
2. The draft invariant "no `DELETE FROM nodes` appears in command code" was too strong once the global `named_entity` rebuild was implemented honestly. The truthful invariant is narrower: per-file replacement still routes through `deleteNodesByFile`, while entity-state rebuild explicitly clears `named_entity` rows as a world-level derived cache reset.
