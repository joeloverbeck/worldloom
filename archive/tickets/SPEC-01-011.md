# SPEC-01-011: Make attribution-target diagnostics truthful after full-world resolution

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — `tools/world-index/src/parse/semantic.ts`, `tools/world-index/src/commands/shared.ts`, and the world-index test suite need a post-resolution attribution diagnostic path instead of the current parse-local warning path.
**Deps**: `archive/tickets/SPEC-01-006.md`, `archive/tickets/SPEC-01-008.md`, `archive/tickets/SPEC-01-009.md`, `docs/FOUNDATIONS.md`, `specs/SPEC-04-validator-framework.md`

## Problem

`world-index build animalia` currently persists `637` `unresolved_attribution_target` warnings even though the corresponding `patched_by` edges resolve correctly after the full build. The live cause is architectural, not corpus-wide corruption: `tools/world-index/src/parse/semantic.ts` emits the warning whenever an attribution target is absent from the current file's parsed node set, while `tools/world-index/src/index/edges.ts` later resolves those same cross-file targets against the full node table. This makes the validation surface noisy and untruthful, which weakens Rule 6 auditability instead of strengthening it.

## Assumption Reassessment (2026-04-22)

1. `tools/world-index/src/parse/semantic.ts` builds `existingNodeIds` from `yamlNodes + proseNodes` passed into `extractSemanticEdges()`. In `tools/world-index/src/commands/shared.ts`, those node arrays are per-file, not world-global, so a domain-file attribution like `<!-- added by CF-0021 -->` is treated as unresolved during parse even though `CF-0021` exists in `CANON_LEDGER.md`.
2. `tools/world-index/src/commands/shared.ts` persists `semantic_edge_extraction` warnings during `insertParsedFile()`, then later calls `resolveUnresolvedEdges()` as part of the full build/sync pipeline. There is no later pass that retracts or recomputes `unresolved_attribution_target` warnings after cross-file resolution succeeds.
3. Live DB evidence from `worlds/animalia/_index/world.db` shows `637` `unresolved_attribution_target` rows concentrated in domain files such as `EVERYDAY_LIFE.md` (`223`) and `ECONOMY_AND_RESOURCES.md` (`101`), while sample `patched_by` edges in those same files already resolve to real CF ids like `CF-0021`, `CF-0024`, and `CF-0028`.
4. FOUNDATIONS Rule 6 (`No Silent Retcons`) and `specs/SPEC-04-validator-framework.md` both require truthful attribution and cross-file reference surfaces. A warning that really means "not present in this one-file parse batch" is weaker than the contract, not stricter.
5. Cross-artifact boundary under audit: attribution comments in prose, `patched_by` / `clarified_by` edges in the index, and `validation_results` rows consumed by build/verify diagnostics.
6. Mismatch + correction: the current warning code name implies a durable orphan reference, but the actual implementation is only a parse-local pre-resolution observation. This ticket should make the persisted diagnostic world-truthful.
7. Integration-proof wrinkle: the checked-in `worlds/animalia/` tree currently includes `_index/index_version.txt`, so the temp-copy integration helper must remove the copied `_index/` directory before asserting a fresh build. That is test-harness setup cleanup, not a change to the attribution contract.

## Architecture Check

1. The clean solution is to distinguish provisional parse-local uncertainty from durable post-resolution failure. That can be done either by delaying the warning until after `resolveUnresolvedEdges()` or by recomputing unresolved attribution targets from persisted edges after the world-level node set exists. Both are cleaner than continuing to store knowingly false warnings.
2. No backwards-compatibility aliasing or duplicate diagnostic codes should be introduced. The end state is one truthful persisted meaning for `unresolved_attribution_target`.

## Verification Layers

1. Cross-file attribution comments that resolve after a full build produce resolved `patched_by` / `clarified_by` edges and no lingering false `unresolved_attribution_target` rows. -> integration test + readonly DB query
2. A genuinely missing attribution target still records one durable warning after full-world resolution. -> fixture command test
3. Incremental sync updates the diagnostic truthfully when a previously missing target is later added. -> command test
4. FOUNDATIONS Rule 6 remains preserved because attribution edges still exist even when the target is missing; only the warning timing/meaning changes. -> FOUNDATIONS alignment check

## What to Change

### 1. Move attribution unresolved diagnostics to a world-truthful phase

Stop persisting `unresolved_attribution_target` during the one-file parse step. Instead, compute and store it only after the full node set exists and `resolveUnresolvedEdges()` has run, or recompute it from persisted unresolved attribution edges in a dedicated post-pass.

### 2. Keep the edge surface unchanged

`patched_by` and `clarified_by` emission should remain exactly as-is. This ticket is about warning truthfulness, not about weakening audit edges.

### 3. Add regression coverage for both false-positive removal and true-positive retention

Prove the real bug with one cross-file fixture and one genuinely missing-target fixture, then prove the end-to-end animalia build no longer produces spurious unresolved attribution warnings.

## Outcome

`unresolved_attribution_target` is no longer persisted during one-file parse. `semantic.ts` still emits `patched_by` and `clarified_by` edges exactly as before, but `shared.ts` now recomputes unresolved attribution diagnostics only after the full world has been indexed and `resolveUnresolvedEdges()` has run.
