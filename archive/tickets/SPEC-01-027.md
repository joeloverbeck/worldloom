# SPEC-01-027: Preserve YAML-derived semantic edges for recovery-parsed canon facts

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Small
**Engine Changes**: Yes — `tools/world-index` semantic edge extraction and its proof surfaces.
**Deps**: `archive/specs/SPEC-01-world-index.md`, `archive/tickets/SPEC-01-006.md`, `archive/tickets/SPEC-01-012.md`, `archive/tickets/SPEC-01-014.md`, `docs/FOUNDATIONS.md`

## Problem

The rebuilt `animalia` DB is structurally clean, but it does not store the full YAML-derived semantic edge set that the live source files author and the current parser contract implies it should. In `worlds/animalia/CANON_LEDGER.md`, the pre-fix rebuilt DB stored only `60` `derived_from` edges, `182` `required_world_update` edges, and `58` `modified_by` edges. A direct recovery-aware replay over the same live `node.body` values showed the authored source contains `87` `derived_from`, `283` `required_world_update`, and `80` `modified_by` edges, while `affected_fact=127` already matched exactly.

This is not a source-data dispute. It is a parser-consistency bug: some Canon Fact YAML blocks are successfully recovered into `canon_fact_record` nodes during Pass 2, but Pass 3 reparses the raw `body` with a stricter path and silently skips all semantic edges for those same records. In the current live `animalia` corpus, this drops every YAML-derived semantic edge from `CF-0002`, `CF-0024`, `CF-0027`, `CF-0028`, `CF-0032`, `CF-0038`, `CF-0041`, and `CF-0043`.

## Assumption Reassessment (2026-04-23)

1. `tools/world-index/src/parse/yaml.ts` currently builds `canon_fact_record` / `change_log_entry` nodes with `parseYamlWithRecovery(rawYaml)` and therefore accepts recoverable ledger YAML that plain `YAML.parse(...)` rejects.
2. `tools/world-index/src/parse/semantic.ts` currently reparses those persisted `node.body` strings with `safeParseYaml(...)` and skips semantic-edge extraction entirely when that plain parse returns `null`.
3. Shared boundary under audit: the handoff from Pass 2 YAML node extraction to Pass 3 YAML-derived edge extraction for `source_basis.derived_from`, `required_world_updates`, and `modification_history[].change_id` in `CANON_LEDGER.md`.
4. `archive/specs/SPEC-01-world-index.md` defines these as first-class YAML-derived edges: `derived_from`, `required_world_update`, `affected_fact`, and `modified_by`. A node that successfully lands as a Canon Fact record should not lose those edges because Pass 3 reuses a weaker parser.
5. `docs/FOUNDATIONS.md` favors structured explicit truth surfaces over silent omission. Dropping authored CF semantic references after the node has already been accepted weakens the canonical audit trail instead of preserving it.
6. Live-corpus evidence from temp-copy rebuilds is concrete: `affected_fact=127` matches the authored Change Log exactly, the pre-fix stored DB lagged the recovery-aware source by `27` `derived_from`, `101` `required_world_update`, and `22` `modified_by` edges, and the post-fix rebuilt DB now matches the authored recovery-aware totals exactly at `derived_from=87`, `required_world_update=283`, `modified_by=80`, and `affected_fact=127`.
7. The affected live source set is bounded. The currently reproduced recovery-only Canon Fact records are `CF-0002`, `CF-0024`, `CF-0027`, `CF-0028`, `CF-0032`, `CF-0038`, `CF-0041`, and `CF-0043`.
8. Adjacent contradictions are out of scope unless they are required consequences of this seam. This ticket is not reopening entity extraction, attribution-comment diagnostics, change-log id salvage, or domain-file target resolution; those are already owned by earlier archived tickets.
9. Mismatch + correction: the rebuilt DB being "structurally clean" does not mean the stored semantic graph is complete. The truthful scope here is semantic-edge completeness for recovery-parsed ledger YAML, not schema repair or world-content repair.

## Architecture Check

1. The clean fix is to make Pass 3 semantic extraction use the same recovery-aware YAML interpretation that already admitted the node in Pass 2, or to transport that parsed object forward explicitly. That is cleaner than keeping two divergent parse contracts over the same persisted `body` text.
2. No backwards-compatibility aliasing/shims introduced.

3. The existing unaffected surface stays truthful: `affected_fact` counts remain unchanged and no new validation noise is introduced for already-clean Change Log parsing. -> targeted integration assertion on live counts.

## What to Change

### 1. Unify semantic-edge parsing with YAML recovery

Update the semantic extraction path so `canon_fact_record` and `change_log_entry` edge extraction does not depend on a second stricter parse of `node.body`. Use the same recovery-aware YAML interpretation that `extractYamlNodes(...)` already trusts for node creation, or pass the recovered structured data forward through a single canonical path.

The resulting semantic contract must preserve all currently supported YAML-derived edges from recovered CF/CH nodes:

- `source_basis.derived_from` -> `derived_from`
- `required_world_updates` -> `required_world_update`
- `modification_history[].change_id` -> `modified_by`
- `affected_fact_ids` -> `affected_fact`

### 2. Prove the live recovered-edge totals

Add or tighten package-local proof so the rebuilt live corpus cannot silently regress this seam again. The proof should name the live recovered-CF examples above and assert the post-build edge totals for `animalia` rather than only checking unresolved targets.
